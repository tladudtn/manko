
COMMANDS

     service-log-level SERVICE [LEVEL]
         LEVEL 인수가 제공되지 않으면, SERVICE가 보고하는 현재 로그 레벨을 출력합니다.

         선택적 인수 LEVEL이 제공되면, 해당 서비스의 현재 로그 레벨을 LEVEL로 변경합니다. 로그 레벨은 일반적인 syslog 로그 레벨이어야 하며, 즉 0...7 범위의 값 또는 emerg, alert, crit, err, warning, notice, info, debug 중 하나여야 합니다. 자세한 내용은 syslog(3)를 참조하십시오.

         해당 서비스는 적절한 BusName=destination 속성을 가지고 있어야 하며, 또한 generic org.freedesktop.LogControl1(5) 인터페이스를 구현해야 합니다. (systemctl은 D-Bus 이름 목적지를 위해 org.freedesktop.LogControl1.LogLevel 인터페이스에 접근하기 위해 generic D-Bus 프로토콜을 사용합니다.)

         버전 247에 추가되었습니다.

     service-log-target SERVICE [TARGET]
         TARGET 인수가 제공되지 않으면, SERVICE가 보고하는 현재 로그 대상을 출력합니다.

         선택적 인수 TARGET이 제공되면, 해당 서비스의 현재 로그 대상을 TARGET으로 변경합니다. 로그 대상은 console (서비스의 표준 오류 스트림으로 로그를 출력), kmsg (커널 로그 버퍼로 로그를 출력), journal (네이티브 journal 프로토콜을 사용하여 systemd-journald.service(8)로 로그를 출력), syslog (클래식 syslog 소켓 /dev/log로 로그를 출력), null (어떠한 로그 출력도 없음) 또는 auto (자동으로 결정된 선택 사항으로, 일반적으로 서비스가 대화형으로 호출된 경우 console과 동일하며, 그렇지 않은 경우 journal 또는 syslog와 동일합니다) 중 하나여야 합니다.

         대부분의 서비스에 대해, 로그 대상의 작은 하위 집합만이 의미가 있습니다. 특히, 대부분의 "일반적인" 서비스는 console, journal, null만 구현해야 합니다. 그 외의 것은 적절한 로깅이 설정되기 전에 매우 초기에 활성화되는 저수준 서비스에만 적합합니다.

         해당 서비스는 적절한 BusName=destination 속성을 가지고 있어야 하며, 또한 generic org.freedesktop.LogControl1(5) 인터페이스를 구현해야 합니다. (systemctl은 D-Bus 이름 목적지를 위해 org.freedesktop.LogControl1.LogLevel 인터페이스에 접근하기 위해 generic D-Bus 프로토콜을 사용합니다.)

         버전 247에 추가되었습니다.

     reset-failed [PATTERN...]
         지정된 유닛의 "failed" 상태를 재설정하거나, 유닛 이름을 전달하지 않으면 모든 유닛의 상태를 재설정합니다. 유닛이 어떤 방식으로든 실패할 때 (예: 0이 아닌 오류 코드로 프로세스가 종료되거나, 비정상적으로 종료되거나, 시간 초과가 발생할 때), 해당 유닛은 자동으로 "failed" 상태로 진입하며, 서비스가 중지/재시작되거나 이 명령어로 재설정될 때까지 관리자가 검사할 수 있도록 종료 코드와 상태가 기록됩니다.

         유닛의 "failed" 상태를 재설정하는 것 외에도, 다양한 다른 유닛별 속성도 재설정합니다. 모든 유닛 유형의 시작 속도 제한 카운터가 0으로 재설정되며, 서비스 유닛의 재시작 카운터도 재설정됩니다. 따라서 유닛의 시작 제한(StartLimitIntervalSec=/StartLimitBurst=로 구성된)에 도달하여 유닛이 다시 시작되는 것을 거부하는 경우, 이 명령어를 사용하여 다시 시작할 수 있도록 합니다.

     whoami [PID...]
         주어진 PID가 참조하는 프로세스가 속한 유닛을 반환합니다 (줄마다 하나씩). PID가 지정되지 않으면 systemctl 명령어가 호출된 유닛을 반환합니다.

         버전 254에 추가되었습니다.

[[[MANKO_HDR_001]]]
     list-unit-files [PATTERN...]
         시스템에 설치된 유닛 파일 목록을 활성화 상태(is-enabled가 보고하는 대로)와 함께 나열합니다. 하나 이상의 PATTERN이 지정되면, 이름이 그중 하나와 일치하는 유닛 파일만 표시됩니다 (유닛 파일 시스템 경로와 일치하는 패턴은 지원되지 않습니다).

         list-units와 달리 이 명령어는 명시적으로 인스턴스화된 유닛 외에도 템플릿 유닛을 나열합니다.

         버전 233에 추가되었습니다.

     enable UNIT..., enable PATH...
         하나 이상의 유닛 또는 유닛 인스턴스를 활성화합니다. 이는 지정된 유닛 파일의 [Install] 섹션에 인코딩된 심볼릭 링크 세트를 생성합니다. 심볼릭 링크가 생성된 후, 변경 사항이 즉시 반영되도록 시스템 관리자 구성이 다시 로드됩니다 (daemon-reload와 동등한 방식으로). 이 작업이 활성화되는 유닛 중 어느 것도 시작하는 효과가 있다는 점에 유의하십시오. 이것이 원하는 경우, 이 명령어를 --now 스위치와 결합하거나 나중에 적절한 인수로 start를 호출하십시오. 유닛 인스턴스 활성화(예: foo@bar.service 형태의 유닛 활성화)의 경우, 인스턴스와 동일한 이름의 심볼릭 링크가 유닛 구성 디렉토리에 생성되지만, 이 링크들은 해당 인스턴스가 인스턴스화된 단일 템플릿 유닛 파일을 가리킵니다.

         이 명령어는 유효한 유닛 이름(이 경우 적절한 이름의 유닛 파일을 위해 다양한 유닛 파일 디렉토리가 자동으로 검색됨) 또는 유닛 파일의 절대 경로(이 경우 해당 파일이 직접 읽힘)를 예상합니다. 지정된 유닛 파일이 일반적인 유닛 파일 디렉토리 외부에 위치하는 경우, 유닛 구성 경로에 연결하는 추가 심볼릭 링크가 생성되어 start와 같은 명령어가 요청할 때 해당 파일이 발견되도록 보장합니다. 연결된 유닛 파일이 위치한 파일 시스템은 systemd가 시작될 때 접근 가능해야 합니다 (예: /home/ 또는 /var/ 아래의 모든 것은 루트 파일 시스템에 해당 디렉토리가 위치하지 않는 한 허용되지 않습니다).

         이 명령어는 실행된 파일 시스템 작업을 출력합니다. 이 출력은 --quiet을 전달하여 억제할 수 있습니다.

         이 작업은 유닛 파일의 [Install] 섹션에 제안된 심볼릭 링크만 생성한다는 점에 유의하십시오. 이 명령어가 유닛 구성 디렉토리를 조작하는 권장 방법이지만, 관리자는 이 디렉토리 아래에 심볼릭 링크를 배치하거나 이동하여 수동으로 추가 변경을 할 수 있습니다. 이는 제안된 기본 설치와 다른 구성을 생성하는 데 특히 유용합니다. 이 경우, 변경 사항이 반영되도록 daemon-reload를 수동으로 호출해야 합니다.

         설치 정보가 없는 유닛에 이 작업을 사용할 때 경고가 표시됩니다. --no-warn을 사용하여 경고를 억제할 수 있습니다.

         유닛 활성화는 start 명령어로 수행되는 유닛 시작(활성화)과 혼동해서는 안 됩니다. 유닛 활성화와 시작은 직교합니다. 유닛은 시작되지 않고 활성화될 수 있으며, 활성화되지 않고 시작될 수도 있습니다. 활성화는 단순히 유닛을 다양한 제안된 위치에 연결합니다 (예를 들어, 유닛이 부팅 시 또는 특정 유형의 하드웨어가 연결될 때 자동으로 시작되도록). 시작은 실제로 데몬 프로세스를 생성하거나 (서비스 유닛의 경우), 소켓을 바인딩하거나 (소켓 유닛의 경우) 등을 수행합니다.

         --system, --user, --runtime 또는 --global이 지정되었는지 여부에 따라, 이는 시스템용, 호출하는 사용자 전용, 시스템의 이번 부팅에만, 또는 모든 사용자의 향후 로그인에 대해 유닛을 활성화합니다. 마지막 경우, 시스템d 데몬 구성은 다시 로드되지 않는다는 점에 유의하십시오.

         마스크된 유닛에 enable을 사용하는 것은 지원되지 않으며 오류가 발생합니다.
