
     soft-reboot
         사용자 공간을 종료하고 재부팅합니다. 이는 systemctl start
         soft-reboot.target --job-mode=replace-irreversibly --no-block과 동일합니다. 이
         명령어는 비동기적입니다. 재부팅 작업이 큐에 추가된 후 반환되며, 완료될 때까지 기다리지 않습니다.

         이 명령어는 halt와 유사한 방식으로 --force 및 --when=을 준수합니다.

         이 작업은 커널을 실행 상태로 유지한 채 사용자 공간만 재부팅합니다. 자세한 내용은
         systemd-soft-reboot.service(8)를 참조하십시오.

         "/run/nextroot/"에 새로운 루트 파일 시스템이 설정된 경우, "SYSTEM‐
         CTL_SKIP_AUTO_SOFT_REBOOT=1"이 설정되지 않는 한 재부팅이 호출될 때 soft-reboot이 수행됩니다.

         버전 254에 추가되었습니다.

     exit [EXIT_CODE]
         서비스 관리자에게 종료를 요청합니다. 이는 사용자 서비스 관리자(즉, --user 옵션과 함께 사용될 때) 또는 컨테이너에서만 지원되며, 그렇지 않은 경우에는 poweroff와 동일합니다. 이 명령어는 비동기적입니다. 종료 작업이 큐에 추가된 후 반환되며, 완료될 때까지 기다리지 않습니다.

         EXIT_CODE가 전달되면 서비스 관리자는 지정된 종료 코드로 종료됩니다.

         버전 227에 추가되었습니다.

     switch-root [ROOT [INIT]]
         다른 루트 디렉터리로 전환하고 그 아래에 새로운 시스템 관리자 프로세스를 실행합니다. 이는 initrd에서 사용하도록 의도되었으며, initrd의 시스템 관리자 프로세스(별칭 "init" 프로세스, PID 1)에서 실제 호스트 루트 파일 시스템에서 로드된 메인 시스템 관리자 프로세스로 전환합니다. 이 호출은 두 개의 인수를 받습니다. 새로운 루트 디렉터리가 될 디렉터리와 PID 1로 실행될 새로운 시스템 관리자 바이너리의 경로입니다. 둘 다 생략되거나 전자가 빈 문자열인 경우 /sysroot/로 기본 설정됩니다. 후자가 생략되거나 빈 문자열인 경우, systemd 바이너리가 자동으로 검색되어 서비스 관리자로 사용됩니다. 시스템 관리자 경로가 생략되거나 빈 문자열과 같거나 systemd 바이너리 경로와 동일한 경우, initrd의 시스템 관리자 프로세스 상태가 메인 시스템 관리자에게 전달되어 initrd 부트 단계에 관련된 서비스 상태를 나중에 검사할 수 있게 합니다.

         버전 209에 추가되었습니다.

     sleep
         시스템을 절전, 최대 절전 모드, 하이브리드 절전 모드 또는 절전 후 최대 절전 모드로 전환합니다. 사용할 절전 작업은 systemd-logind.service(8)에 의해 자동으로 선택됩니다. 기본적으로 suspend-then-hibernate가 사용되며, 지원되지 않는 경우 suspend로 폴백하고 그 다음 hibernate합니다. 자세한 내용은 logind.conf(5)의 SleepOperation= 설정을 참조하십시오. 이 명령어는 비동기적이며, 절전 작업이 성공적으로 큐에 추가된 후 반환됩니다. 절전/재개 주기가 완료될 때까지 기다리지 않습니다.

         버전 256에 추가되었습니다.

     suspend
         시스템을 절전 상태로 전환합니다. 이는 특별한 타겟 유닛 suspend.target의 활성화를 유발합니다. 이 명령어는 비동기적이며, 절전 작업이 성공적으로 큐에 추가된 후 반환됩니다. 절전/재개 주기가 완료될 때까지 기다리지 않습니다.

         --force가 지정되었고 systemd-logind가 해당 작업에 대해 오류를 반환한 경우, 오류는 무시되고 타겟 유닛을 직접 시작하여 작업이 다시 시도됩니다.

     hibernate
         시스템을 최대 절전 모드로 전환합니다. 이는 특별한 타겟 유닛 hibernate.target의 활성화를 유발합니다. 이 명령어는 비동기적이며, 최대 절전 모드 작업이 성공적으로 큐에 추가된 후 반환됩니다. 최대 절전 모드/복원 주기가 완료될 때까지 기다리지 않습니다.

         이 명령어는 suspend와 동일한 방식으로 --force를 준수합니다.

     hybrid-sleep
         시스템을 최대 절전 모드 및 절전 상태로 전환합니다. 이는 특별한 타겟 유닛 hybrid-sleep.target의 활성화를 유발합니다. 이 명령어는 비동기적이며, 하이브리드 절전 작업이 성공적으로 큐에 추가된 후 반환됩니다. 절전/깨우기 주기가 완료될 때까지 기다리지 않습니다.

         이 명령어는 suspend와 동일한 방식으로 --force를 준수합니다.

         버전 196에 추가되었습니다.

     suspend-then-hibernate
         배터리가 부족하거나 systemd-sleep.conf에 지정된 지연 시간이 경과했을 때 시스템을 절전 상태로 전환하고 최대 절전 모드로 전환합니다. 이는 특별한 타겟 유닛 suspend-then-hibernate.target의 활성화를 유발합니다. 이 명령어는 비동기적이며, 하이브리드 절전 작업이 성공적으로 큐에 추가된 후 반환됩니다. 절전/깨우기 또는 최대 절전 모드/복원 주기가 완료될 때까지 기다리지 않습니다.

         이 명령어는 suspend와 동일한 방식으로 --force를 준수합니다.

         버전 240에 추가되었습니다.

   Parameter Syntax
     위에 나열된 유닛 명령어는 단일 유닛 이름(UNIT으로 지정됨) 또는 여러 유닛 사양(PATTERN...으로 지정됨)을 사용합니다. 첫 번째 경우, 접미사가 있든 없든 유닛 이름을 지정해야 합니다. 접미사가 지정되지 않은 경우(유닛 이름이 "약어"인 경우), systemctl은 적절한 접미사(기본적으로 ".service")를 추가하며, 특정 유닛 유형에만 작동하는 명령어의 경우 유형별 접미사를 추가합니다. 예를 들어,

         # systemctl start sshd

     와

         # systemctl start sshd.service

     는 동일하며,

         # systemctl isolate default

     와

         # systemctl isolate default.target

     도 동일합니다.

     장치 노드에 대한 (절대) 경로는 자동으로 장치 유닛 이름으로 변환되며, 마운트 유닛 이름에 대한 다른 (절대) 경로는 변환됩니다.

         # systemctl status /dev/sda
         # systemctl status /home

     는 다음과 동일합니다.

         # systemctl status dev-sda.device
         # systemctl status home.mount

     두 번째 경우, 셸 스타일의 글롭은 현재 메모리에 있는 모든 유닛의 기본 이름과 일치합니다. 접미사가 있든 없든 리터럴 유닛 이름은 첫 번째 경우와 동일하게 처리됩니다. 이는 리터럴 유닛 이름이 항상 정확히 하나의 유닛을 참조한다는 것을 의미하지만, 글롭은 0개의 유닛과 일치할 수 있으며 이는 오류로 간주되지 않습니다.

     글롭 패턴은 fnmatch(3)을 사용하므로 일반적인 셸 스타일 글롭 규칙이 사용되며, "*" , "?" , "[]"를 사용할 수 있습니다. 자세한 내용은 glob(7)을 참조하십시오. 패턴은 현재 메모리에 있는 유닛의 기본 이름과 일치하며, 아무것도 일치하지 않는 패턴은 조용히 건너뜁니다. 예를 들어,

         # systemctl stop "sshd@*.service"

     는 모든 sshd@.service 인스턴스를 중지합니다. 유닛의 별칭 이름과 메모리에 없는 유닛은 글롭 확장에 고려되지 않는다는 점에 유의하십시오.

     유닛 파일 명령어의 경우, 지정된 UNIT은 유닛 파일의 이름(가능하게는 약어, 위 참조) 또는 유닛 파일의 절대 경로여야 합니다.

         # systemctl enable foo.service

     또는

         # systemctl link /path/to/foo.service
