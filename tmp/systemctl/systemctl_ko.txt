SYSTEMCTL(1)                       systemctl                       SYSTEMCTL(1)

NAME
     systemctl - systemd 시스템 및 서비스 관리자 제어

SYNOPSIS
     systemctl [OPTIONS...] COMMAND [UNIT...]

DESCRIPTION
     systemctl은 "systemd" 시스템 및 서비스 관리자의 상태를 검사하고 제어하는 데 사용될 수 있습니다. 이 도구가 관리하는 기본 개념 및 기능에 대한 소개는 systemd(1)을 참조하십시오.

COMMANDS
     다음 명령이 이해됩니다:

   Unit Commands (Introspection and Modification)
     list-units [PATTERN...]
         systemd가 현재 메모리에 가지고 있는 유닛을 나열합니다. 여기에는 직접 참조되거나 종속성을 통해 참조되는 유닛, 애플리케이션에 의해 프로그래밍 방식으로 고정된 유닛, 또는 과거에 활성화되었으나 실패한 유닛이 포함됩니다. 기본적으로 활성화되었거나, 보류 중인 작업을 가지고 있거나, 실패한 유닛만 표시되며, 이는 --all 옵션으로 변경할 수 있습니다. 하나 이상의 PATTERN이 지정되면, 그중 하나와 일치하는 유닛만 표시됩니다. 표시되는 유닛은 해당 옵션이 지정된 경우 --type= 및 --state=에 의해 추가로 필터링됩니다.

         이 명령어는 유닛 템플릿을 표시하지 않고, 유닛 템플릿의 인스턴스만 표시한다는 점에 유의하십시오. 인스턴스화되지 않은 유닛 템플릿은 실행할 수 없으므로 이 명령어의 출력에 절대 나타나지 않습니다. 구체적으로 이는 foo@.service가 인스턴스화되지 않는 한 (예: foo@bar.service로) 이 목록에 나타나지 않음을 의미합니다. 설치된 유닛 템플릿 파일을 나열하려면 list-unit-files(아래 참조)를 사용하십시오.

         다음과 유사한 출력을 생성합니다:

               UNIT                         LOAD   ACTIVE SUB     DESCRIPTION
               sys-module-fuse.device       loaded active plugged /sys/module/fuse
               -.mount                      loaded active mounted Root Mount
               boot-efi.mount               loaded active mounted /boot/efi
               systemd-journald.service     loaded active running Journal Service
               systemd-logind.service       loaded active running Login Service
             ● user@1000.service            loaded failed failed  User Manager for UID 1000
               ...
               systemd-tmpfiles-clean.timer loaded active waiting Daily Cleanup of Temporary Directories

             LOAD   = 유닛 정의가 올바르게 로드되었는지 여부를 반영합니다.
             ACTIVE = 높은 수준의 유닛 활성화 상태, 즉 SUB의 일반화입니다.
             SUB    = 낮은 수준의 유닛 활성화 상태이며, 값은 유닛 유형에 따라 다릅니다.

             123개의 로드된 유닛이 나열되었습니다. --all을 사용하여 로드되었지만 비활성인 유닛도 확인하십시오.
             모든 설치된 유닛 파일을 표시하려면 'systemctl list-unit-files'를 사용하십시오.

         터미널이 지원하는 경우 헤더와 주어진 유형의 마지막 유닛은 밑줄이 그어집니다. 마스킹되었거나, 발견되지 않았거나, 기타 이유로 실패한 서비스 옆에는 색상 점이 표시됩니다.

         LOAD 열은 로드 상태를 보여주며, loaded, not-found, bad-setting, error, masked 중 하나입니다. ACTIVE 열은 일반적인 유닛 상태를 보여주며, 다음 중 하나입니다:

         Table 1. Unit ACTIVE 상태
         ┌──────────────┬────────────────────────────┐
         │ State        │ Description                │
         ├──────────────┼────────────────────────────┤
         │ active       │ 시작됨, 바인딩됨, 플러그됨 │
         │              │ in, ..., 유닛 유형에 따라 다름. │
         ├──────────────┼────────────────────────────┤
         │ inactive     │ 중지됨, 바인딩 해제됨, 플러그 해제됨, ..., 유닛 유형에 따라 다름. │
         ├──────────────┼────────────────────────────┤
         │ failed       │ inactive와 유사하지만, 유닛이 어떤 방식으로든 실패한 경우 (프로세스가 종료 시 오류 코드를 반환, 충돌, 작업 시간 초과 또는 너무 많은 재시작 후). │
         ├──────────────┼────────────────────────────┤
         │ activating   │ inactive에서 active로 변경 중.                    │
         ├──────────────┼────────────────────────────┤
         │ deactivating │ active에서 inactive로 변경 중.                  │
         ├──────────────┼────────────────────────────┤
         │ maintenance  │ 유닛이 inactive 상태이며 유지보수 작업이 진행 중인 경우.               │
         ├──────────────┼────────────────────────────┤
         │ reloading    │ 유닛이 active 상태이며 구성 파일을 다시 로드 중인 경우.                      │
         ├──────────────┼────────────────────────────┤
         │ refreshing   │ 유닛이 active 상태이며 네임스페이스 내에서 새 마운트가 활성화 중인 경우.          │
         └──────────────┴────────────────────────────┘

         SUB 열은 유닛의 유닛 유형별 상세 상태를 보여주며, 가능한 값은 유닛 유형에 따라 다릅니다. 가능한 LOAD, ACTIVE, SUB 상태 목록은 고정되어 있지 않으며, 새로운 systemd 릴리스에서는 값을 추가하거나 제거할 수 있습니다.

             systemctl --state=help

         이 명령은 현재 가능한 값 세트를 표시하는 데 사용될 수 있습니다.

         이것이 기본 명령어입니다.

     list-automounts [PATTERN...]
         현재 메모리에 있는 자동 마운트 유닛을 마운트 경로 순서로 나열합니다. 하나 이상의 PATTERN이 지정되면, 그중 하나와 일치하는 자동 마운트 유닛만 표시됩니다. 다음과 유사한 출력을 생성합니다:

             WHAT        WHERE                    MOUNTED IDLE TIMEOUT UNIT
             /dev/sdb1   /mnt/test                no      120s         mnt-test.automount
             binfmt_misc /proc/sys/fs/binfmt_misc yes     0            proc-sys-fs-binfmt_misc.automount

             2개의 자동 마운트가 나열되었습니다.

         또한 --show-types, --all, 및 --state=를 참조하십시오.

         버전 252에 추가되었습니다.

     list-paths [PATTERN...]
         현재 메모리에 있는 경로 유닛을 경로 순서로 나열합니다. 하나 이상의 PATTERN이 지정되면, 그중 하나와 일치하는 경로 유닛만 표시됩니다. 다음과 유사한 출력을 생성합니다:

             PATH                           CONDITION         UNIT                               ACTIVATES
             /run/systemd/ask-password      DirectoryNotEmpty systemd-ask-password-plymouth.path systemd-ask-password-plymouth.service
             /run/systemd/ask-password      DirectoryNotEmpty systemd-ask-password-wall.path     systemd-ask-password-wall.service
             /var/cache/cups/org.cups.cupsd PathExists        cups.path                          cups.service

             3개의 경로가 나열되었습니다.

         또한 --show-types, --all, 및 --state=를 참조하십시오.

         버전 254에 추가되었습니다.

     list-sockets [PATTERN...]
         현재 메모리에 있는 소켓 유닛을 리스닝 주소 순서로 나열합니다. 하나 이상의 PATTERN이 지정되면, 그중 하나와 일치하는 소켓 유닛만 표시됩니다. 다음과 유사한 출력을 생성합니다:

             LISTEN           UNIT                        ACTIVATES
             kobject-uevent 1 systemd-udevd-kernel.socket systemd-udevd.service
             /dev/rfkill      systemd-rfkill.socket       systemd-rfkill.service
             ...

             5개의 소켓이 나열되었습니다.

         참고: 주소에 공백이 포함될 수 있으므로 이 출력은 프로그램적 소비에 적합하지 않습니다.

         또한 --show-types, --all, 및 --state=를 참조하십시오.

         버전 202에 추가되었습니다.

     list-timers [PATTERN...]
         현재 메모리에 있는 타이머 유닛을 다음 만료 시간 순서로 나열합니다. 하나 이상의 PATTERN이 지정된 경우, 그중 하나와 일치하는 유닛만 표시됩니다. 다음과 유사한 출력을 생성합니다.

             NEXT                         LEFT          LAST                         PASSED     UNIT                         ACTIVATES
             -                            -             Thu 2017-02-23 13:40:29 EST  3 days ago ureadahead-stop.timer        ureadahead-stop.service
             Sun 2017-02-26 18:55:42 EST  1min 14s left Thu 2017-02-23 13:54:44 EST  3 days ago systemd-tmpfiles-clean.timer systemd-tmpfiles-clean.service
             Sun 2017-02-26 20:37:16 EST  1h 42min left Sun 2017-02-26 11:56:36 EST  6h ago     apt-daily.timer              apt-daily.service
             Sun 2017-02-26 20:57:49 EST  2h 3min left  Sun 2017-02-26 11:56:36 EST  6h ago     snapd.refresh.timer          snapd.refresh.service

         NEXT는 타이머가 다음에 실행될 시간을 보여줍니다.

         LEFT는 타이머가 다음에 실행될 때까지 남은 시간을 보여줍니다.

         LAST는 타이머가 마지막으로 실행된 시간을 보여줍니다.

         PASSED는 타이머가 마지막으로 실행된 이후 경과된 시간을 보여줍니다.

         UNIT은 타이머의 이름을 보여줍니다.

         ACTIVATES는 타이머가 실행될 때 활성화하는 서비스의 이름을 보여줍니다.

         --all 및 --state=.도 참조하십시오.

         버전 209에 추가되었습니다.

     is-active PATTERN...
         지정된 유닛 중 하나라도 활성 상태인지(즉, 실행 중인지) 확인합니다.
         하나 이상이 활성 상태이면 종료 코드 0을 반환하고, 그렇지 않으면 0이 아닌 값을 반환합니다. --quiet가 지정되지 않은 경우, 현재 유닛 상태를 표준 출력에도 출력합니다.

     is-failed [PATTERN...]
         지정된 유닛 중 하나라도 "failed" 상태인지 확인합니다. 유닛이 지정되지 않은 경우, is-system-running이 반환하는 "degraded" 상태에 해당하는 실패한 유닛이나 순서 사이클이 있는지 확인합니다. 하나 이상이 실패했으면 종료 코드 0을 반환하고, 그렇지 않으면 0이 아닌 값을 반환합니다. --quiet가 지정되지 않은 경우, 현재 유닛 또는 시스템 상태를 표준 출력에도 출력합니다.

         버전 197에 추가되었습니다.

     status [PATTERN...|PID...]]
         전체 시스템 또는 하나 이상의 유닛에 대한 런타임 상태 정보와 가장 최근의 journal 로그 데이터를 표시합니다. 위치 인수가 지정되지 않았고, --type=, --state= 또는 --failed를 사용하여 유닛 필터가 제공되지 않으면, 전체 시스템의 상태를 표시합니다.
         --all과 결합되면, 모든 유닛의 상태가 뒤따릅니다. 위치 인수가 지정된 경우, 각 위치 인수는 표시할 유닛 이름, 해당 패턴과 이름이 일치하는 유닛을 표시하기 위한 glob 패턴, 또는 해당 PID를 포함하는 유닛을 표시하기 위한 PID 중 하나로 처리됩니다. --type=, --state= 또는 --failed를 사용할 때, 유닛은 TYPE 및 ACTIVE 상태로 추가 필터링됩니다.

         이 함수는 사람이 읽을 수 있는 출력을 생성하는 데 사용됩니다. 컴퓨터가 파싱할 수 있는 출력을 찾고 있다면 대신 show를 사용하십시오. 기본적으로 이 함수는 10줄의 출력만 표시하며 터미널 창에 맞게 줄을 생략합니다. 이는 위에서 본 --lines 및 --full로 변경할 수 있습니다. 또한, journalctl --unit=NAME 또는 journalctl --user-unit=NAME은 메시지에 대해 유사한 필터를 사용하며 더 편리할 수 있습니다.

         이 작업은 런타임 상태만 표시한다는 점에 유의하십시오. 즉, 유닛의 현재 호출(실행 중인 경우) 또는 가장 최근 호출(더 이상 실행 중이 아니며 메모리에서 해제되지 않은 경우)에 대한 정보입니다. 이전 호출, 이전 시스템 부팅에서의 호출 또는 이미 메모리에서 해제된 이전 호출에 대한 정보는 journalctl --unit=.을 통해 검색할 수 있습니다.

         systemd는 필요한 경우 유닛을 암묵적으로 로드하므로, status를 실행하는 것만으로는 파일을 로드하려고 시도합니다. 따라서 이 명령어는 무언가가 이미 로드되었는지 여부를 판단하는 데 유용하지 않습니다. 이후 메모리에 유지할 이유가 없다면, 유닛은 작업 완료 후 빠르게 언로드될 수도 있습니다.

         예시 1. systemctl status의 예시 출력

             $ systemctl status bluetooth
             ● bluetooth.service - Bluetooth service
                Loaded: loaded (/usr/lib/systemd/system/bluetooth.service; enabled; preset: enabled)
                Active: active (running) since Wed 2017-01-04 13:54:04 EST; 1 weeks 0 days ago
                  Docs: man:bluetoothd(8)
              Main PID: 930 (bluetoothd)
                Status: "Running"
                 Tasks: 1
                Memory: 648.0K
                   CPU: 435ms
                CGroup: /system.slice/bluetooth.service
                        └─930 /usr/lib/bluetooth/bluetoothd

             Jan 12 10:46:45 example.com bluetoothd[8900]: Not enough free handles to register service
             Jan 12 10:46:45 example.com bluetoothd[8900]: Current Time Service could not be registered
             Jan 12 10:46:45 example.com bluetoothd[8900]: gatt-time-server: Input/output error (5)

         점 ("●")은 지원되는 터미널에서 유닛 상태를 한눈에 요약하기 위해 색상을 사용합니다. 색상과 함께 모양은 상태에 따라 달라집니다. "inactive" 또는 "maintenance"는 흰색 원 ("○"), "active"는 녹색 점 ("●"), "deactivating"은 흰색 점, "failed" 또는 "error"는 빨간색 십자 ("×"), "reloading" 또는 "refreshing"은 녹색 시계 방향 원형 화살표 ("↻")입니다.

         출력의 "Loaded:" 줄은 유닛이 메모리에 로드된 경우 "loaded"를 표시합니다. "Loaded:"의 다른 가능한 값에는 로드하는 데 문제가 있었을 경우 "error", 이 유닛에 대한 유닛 파일을 찾지 못한 경우 "not-found", 필수 유닛 파일 설정을 구문 분석할 수 없었던 경우 "bad-setting", 유닛 파일이 마스킹된 경우 "masked"가 포함됩니다. 이 줄은 유닛 파일의 경로를 표시하는 것 외에도 활성화 상태를 표시합니다. 활성화된 유닛은 유닛 간의 종속성 네트워크에 포함되며, 따라서 부팅 시 또는 다른 형태의 활성화를 통해 시작됩니다. "masked"의 정의를 포함하여 가능한 모든 활성화 상태의 전체 표는 is-enabled 명령어의 문서를 참조하십시오.

         "Active:" 줄은 활성 상태를 표시합니다. 값은 일반적으로 "active" 또는 "inactive"입니다. Active는 유닛 유형에 따라 시작됨, 바인딩됨, 플러그인됨 등을 의미할 수 있습니다. 유닛은 상태를 변경하는 과정에 있을 수도 있으며, "activating" 또는 "deactivating" 상태를 보고할 수 있습니다. 서비스가 충돌, 오류 코드로 종료되거나 시간 초과와 같이 어떤 방식으로든 실패하면 특별한 "failed" 상태가 진입됩니다. 실패 상태가 진입되면 나중에 참조할 수 있도록 원인이 기록됩니다.

     show [PATTERN...|JOB...]
         하나 이상의 유닛, 작업 또는 매니저 자체의 속성을 표시합니다. 인수가 지정되지 않으면 매니저의 속성이 표시됩니다. 유닛 이름이 지정되면 유닛의 속성이 표시되고, 작업 ID가 지정되면 작업의 속성이 표시됩니다. 기본적으로 빈 속성은 억제됩니다. --all을 사용하여 해당 속성도 표시하십시오. 표시할 특정 속성을 선택하려면 --property=.를 사용하십시오. 이 명령어는 컴퓨터가 파싱할 수 있는 출력이 필요할 때 사용하도록 의도되었습니다. 형식화된 사람이 읽을 수 있는 출력을 찾고 있다면 status를 사용하십시오.

         systemctl show가 표시하는 많은 속성은 시스템 및 서비스 매니저와 그 유닛 파일의 구성 설정에 직접 매핑됩니다. 이 명령어에 의해 표시되는 속성은 일반적으로 원래 구성 설정의 더 낮은 수준의 정규화된 버전이며 구성 외에 런타임 상태를 노출한다는 점에 유의하십시오. 예를 들어, 서비스 유닛에 대해 표시되는 속성에는 서비스의 현재 메인 프로세스 식별자("MainPID"(런타임 상태))가 포함되며, 마이크로초는 시스템 및 서비스 매니저가 내부적으로 사용하는 정규화된 시간 단위이기 때문에 일치하는 구성 옵션이 "...Sec"으로 끝나는 경우에도 시간 설정은 항상 "...USec" 접미사로 끝나는 속성으로 노출됩니다.

         이러한 많은 속성에 대한 자세한 내용은 이 속성을 지원하는 D-Bus 인터페이스의 문서를 참조하십시오. org.freedesktop.systemd1(5)를 참조하십시오.

     cat PATTERN...

     list-dependencies [UNIT...]
         지정된 유닛이 필요로 하거나 원하는 유닛을 표시합니다. 이 명령어는 Requires=, Requisite=, Wants=, ConsistsOf=, BindsTo=, Upholds= 의존성을 따라 유닛을 재귀적으로 나열합니다. 유닛이 지정되지 않은 경우 default.target이 암시됩니다.

         표시되는 유닛은 해당 옵션이 지정된 경우 --type= 및 --state=에 의해 추가로 필터링됩니다. 이 경우 트리 구조를 사용할 수 없으므로 --plain이 암시됩니다.

         기본적으로 대상 유닛만 재귀적으로 확장됩니다. --all이 전달되면 다른 모든 유닛도 재귀적으로 확장됩니다.

         옵션 --reverse, --after, --before를 사용하여 표시되는 의존성의 유형을 변경할 수 있습니다.

         이 명령어는 서비스 관리자에 의해 현재 메모리에 로드된 유닛만 나열한다는 점에 유의하십시오. 특히, 이 명령어는 현재 로드되지 않은 유닛이 선언한 의존성을 나열하지 않기 때문에 특정 유닛에 대한 모든 역 의존성 목록을 포괄적으로 얻는 데 적합하지 않습니다.

         버전 198에 추가되었습니다.

     start PATTERN...
         명령줄에 지정된 하나 이상의 유닛을 시작(활성화)합니다.

         유닛 글롭 패턴은 현재 메모리에 있는 유닛 이름으로 확장된다는 점에 유의하십시오. 활성화되어 있지 않고 실패 상태가 아닌 유닛은 일반적으로 메모리에 없으며 어떤 패턴으로도 일치하지 않습니다. 또한, 인스턴스화된 유닛의 경우 systemd는 인스턴스가 시작될 때까지 인스턴스 이름을 알지 못하는 경우가 많습니다. 따라서 start와 글롭 패턴을 사용하는 것은 유용성이 제한적입니다. 또한, 유닛의 보조 별칭 이름은 고려되지 않습니다.

         옵션 --all은 다른 로드된 유닛에 의해 참조되는 비활성 유닛에 대해서도 작동하도록 사용할 수 있습니다. 이는 이전 단락에서 설명했듯이 그러한 목록이 정의되지 않기 때문에 "모든" 가능한 유닛에 대해 작동하는 것과는 같지 않다는 점에 유의하십시오. 그럼에도 불구하고, 패턴과 일치해야 하는 모든 유닛이 로드된 것으로 알려진 특정 대상을 통해 끌어와지는 경우 systemctl start --all GLOB이 유용할 수 있습니다.

     stop PATTERN...
         명령줄에 지정된 하나 이상의 유닛을 중지(비활성화)합니다.

         유닛이 존재하지 않거나 유닛 중지가 금지된 경우(systemd.unit(5)의 RefuseManualStop= 참조) 이 명령어는 실패합니다. 유닛을 중지하도록 구성된 명령(ExecStop= 등) 중 하나가 실패하더라도 실패하지 않습니다. 왜냐하면 관리자가 여전히 유닛을 강제로 종료할 것이기 때문입니다.

         중지되는 유닛이 다른 유닛에 의해 여전히 트리거될 수 있는 경우, 트리거하는 유닛의 이름을 포함하는 경고가 표시됩니다. --no-warn을 사용하여 경고를 억제할 수 있습니다.

     reload PATTERN...
         명령줄에 나열된 모든 유닛에게 구성을 다시 로드하도록 요청합니다. 이는 서비스별 구성을 다시 로드하는 것이지 systemd의 유닛 구성 파일을 다시 로드하는 것이 아니라는 점에 유의하십시오. systemd가 유닛의 구성 파일을 다시 로드하도록 하려면 daemon-reload 명령을 사용하십시오. 다시 말해: Apache의 예시의 경우, 이는 웹 서버의 Apache의 httpd.conf를 다시 로드하는 것이지 apache.service systemd 유닛 파일은 아닙니다.

         이 명령어는 daemon-reload 명령어와 혼동해서는 안 됩니다.

     restart PATTERN...
         명령줄에 지정된 하나 이상의 유닛을 중지한 다음 시작합니다. 유닛이 아직 실행 중이 아니라면 시작됩니다.

         이 명령으로 유닛을 다시 시작하는 것이 유닛이 다시 시작되기 전에 유닛의 모든 리소스를 반드시 플러시하는 것은 아니라는 점에 유의하십시오. 예를 들어, 서비스별 파일 디스크립터 저장소 시설(systemd.service(5)의 FileDescriptorStoreMax= 참조)은 유닛에 보류 중인 작업이 있는 한 그대로 유지되며, 유닛이 완전히 중지되고 더 이상 보류 중인 작업이 없을 때만 지워집니다. 만약 파일 디스크립터 저장소도 재시작 작업 중에 플러시되기를 의도한다면, 명시적인 systemctl stop 명령 다음에 systemctl start 명령을 발행해야 합니다.

     try-restart PATTERN...
         유닛이 실행 중인 경우 명령줄에 지정된 하나 이상의 유닛을 중지한 다음 시작합니다. 유닛이 실행 중이지 않으면 아무 작업도 수행하지 않습니다.

     enqueue-marked
         해당 "needs-*" 마커가 설정된 모든 유닛에 대해 시작/중지/재시작/재로드 작업을 예약합니다. 재로드를 위해 표시된 유닛이 재로드를 지원하지 않는 경우, 재시작이 예약됩니다. 이러한 속성은 set-property Markers=....를 사용하여 설정할 수 있습니다.

         --no-block을 사용하지 않는 한, systemctl은 예약된 작업이 완료될 때까지 기다립니다.

     isolate UNIT

     set-property UNIT PROPERTY=VALUE...
         지원되는 경우 런타임에 지정된 단위 속성을 설정합니다.

         지정된 단위가 비활성 상태인 것으로 보이면, 변경 사항은 이전에 설명한 대로 디스크에만 저장되므로 단위가 시작될 때 유효하게 됩니다.

         이 명령은 여러 속성을 동시에 변경할 수 있도록 허용하며, 개별적으로 설정하는 것보다 선호됩니다.

         예시: systemctl set-property foobar.service CPUWeight=200 Memory‐Max=2G IPAccounting=yes

         단위 파일 구성 설정과 마찬가지로, 빈 설정을 할당하면 일반적으로 속성이 기본값으로 재설정됩니다.

         예시: systemctl set-property avahi-daemon.service IPAddressDeny=

     bind UNIT PATH [PATH]
         호스트의 파일 또는 디렉토리를 지정된 단위의 마운트 네임스페이스로 바인드 마운트합니다. 첫 번째 경로 인수는 호스트의 소스 파일 또는 디렉토리이며, 두 번째 경로 인수는 단위의 마운트 네임스페이스 내의 대상 파일 또는 디렉토리입니다. 후자가 생략되면, 단위의 마운트 네임스페이스 내의 대상 경로는 호스트의 소스 경로와 동일합니다. --read-only 스위치와 결합되면 읽기 전용 바인드 마운트가 생성됩니다. --mkdir 스위치와 결합되면 마운트가 적용되기 전에 대상 경로가 먼저 생성됩니다.

         이 옵션은 현재 마운트 네임스페이스 내에서 실행되는 단위에만 지원된다는 점에 유의하십시오 (예: RootImage=, PrivateMounts= 등). 이 명령은 디렉토리, 일반 파일, 장치 노드, AF_UNIX 소켓 노드 및 FIFO의 바인드 마운트를 지원합니다. 바인드 마운트는 일시적이며, 현재 단위 프로세스가 존재하는 즉시 해제됩니다. 여기서 바인드 마운트가 추가될 네임스페이스는 메인 서비스 프로세스가 실행되는 네임스페이스라는 점에 유의하십시오. 다른 프로세스(ExecReload=, ExecStartPre= 등에서 실행되는 프로세스)는 별도의 네임스페이스에서 실행됩니다.

         커널에서 지원되는 경우, 선택된 대상에 대한 이전 마운트는 새 마운트로 대체됩니다. 지원되지 않는 경우, 이전 마운트는 덮어 마운트되지만 고정되고 접근할 수 없게 유지됩니다.

         버전 248에 추가되었습니다.

     mount-image UNIT IMAGE [PATH [PARTITION_NAME:MOUNT_OPTIONS]]
         호스트의 이미지를 지정된 단위의 마운트 네임스페이스로 마운트합니다. 첫 번째 경로 인수는 호스트의 소스 이미지이며, 두 번째 경로 인수는 단위의 마운트 네임스페이스 내의 대상 디렉토리입니다 (즉, RootImage=/RootDirectory= 내부). 다음 인수는 있다면, 해당 파티션의 파티션 이름과 쉼표로 구분된 마운트 옵션 목록의 콜론으로 구분된 튜플로 해석됩니다. 형식은 서비스 MountImages= 설정과 동일합니다. --read-only 스위치와 결합되면 읽기 전용 마운트가 생성됩니다. --mkdir 스위치와 결합되면 마운트가 적용되기 전에 대상 경로가 먼저 생성됩니다.

         이 옵션은 현재 마운트 네임스페이스 내에서 실행되는 단위에만 지원된다는 점에 유의하십시오 (즉, RootImage=, PrivateMounts= 등). 이미지 마운트가 추가될 여기서 언급된 네임스페이스는 메인 서비스 프로세스가 실행되는 네임스페이스라는 점에 유의하십시오. 바인드 마운트가 추가될 여기서 언급된 네임스페이스는 메인 서비스 프로세스가 실행되는 네임스페이스라는 점에 유의하십시오. 다른 프로세스(ExecReload=, ExecStartPre= 등에서 실행되는 프로세스)는 별도의 네임스페이스에서 실행됩니다.

         커널에서 지원되는 경우, 선택된 대상에 대한 이전 마운트는 새 마운트로 대체됩니다. 지원되지 않는 경우, 이전 마운트는 덮어 마운트되지만 고정되고 접근할 수 없게 유지됩니다.

         예시:

             systemctl mount-image foo.service /tmp/img.raw /var/lib/image root:ro,nosuid

             systemctl mount-image --mkdir bar.service /tmp/img.raw /var/lib/baz/img

     service-log-level SERVICE [LEVEL]
         LEVEL 인수가 제공되지 않으면, SERVICE가 보고하는 현재 로그 레벨을 출력합니다.

         선택적 인수 LEVEL이 제공되면, 해당 서비스의 현재 로그 레벨을 LEVEL로 변경합니다. 로그 레벨은 일반적인 syslog 로그 레벨이어야 하며, 즉 0...7 범위의 값 또는 emerg, alert, crit, err, warning, notice, info, debug 중 하나여야 합니다. 자세한 내용은 syslog(3)를 참조하십시오.

         해당 서비스는 적절한 BusName=destination 속성을 가지고 있어야 하며, 또한 generic org.freedesktop.LogControl1(5) 인터페이스를 구현해야 합니다. (systemctl은 D-Bus 이름 목적지를 위해 org.freedesktop.LogControl1.LogLevel 인터페이스에 접근하기 위해 generic D-Bus 프로토콜을 사용합니다.)

         버전 247에 추가되었습니다.

     service-log-target SERVICE [TARGET]
         TARGET 인수가 제공되지 않으면, SERVICE가 보고하는 현재 로그 대상을 출력합니다.

         선택적 인수 TARGET이 제공되면, 해당 서비스의 현재 로그 대상을 TARGET으로 변경합니다. 로그 대상은 console (서비스의 표준 오류 스트림으로 로그를 출력), kmsg (커널 로그 버퍼로 로그를 출력), journal (네이티브 journal 프로토콜을 사용하여 systemd-journald.service(8)로 로그를 출력), syslog (클래식 syslog 소켓 /dev/log로 로그를 출력), null (어떠한 로그 출력도 없음) 또는 auto (자동으로 결정된 선택 사항으로, 일반적으로 서비스가 대화형으로 호출된 경우 console과 동일하며, 그렇지 않은 경우 journal 또는 syslog와 동일합니다) 중 하나여야 합니다.

         대부분의 서비스에 대해, 로그 대상의 작은 하위 집합만이 의미가 있습니다. 특히, 대부분의 "일반적인" 서비스는 console, journal, null만 구현해야 합니다. 그 외의 것은 적절한 로깅이 설정되기 전에 매우 초기에 활성화되는 저수준 서비스에만 적합합니다.

         해당 서비스는 적절한 BusName=destination 속성을 가지고 있어야 하며, 또한 generic org.freedesktop.LogControl1(5) 인터페이스를 구현해야 합니다. (systemctl은 D-Bus 이름 목적지를 위해 org.freedesktop.LogControl1.LogLevel 인터페이스에 접근하기 위해 generic D-Bus 프로토콜을 사용합니다.)

     reset-failed [PATTERN...]
         지정된 유닛의 "failed" 상태를 재설정하거나, 유닛 이름을 전달하지 않으면 모든 유닛의 상태를 재설정합니다. 유닛이 어떤 방식으로든 실패할 때 (예: 0이 아닌 오류 코드로 프로세스가 종료되거나, 비정상적으로 종료되거나, 시간 초과가 발생할 때), 해당 유닛은 자동으로 "failed" 상태로 진입하며, 서비스가 중지/재시작되거나 이 명령어로 재설정될 때까지 관리자가 검사할 수 있도록 종료 코드와 상태가 기록됩니다.

   Unit File Commands
     list-unit-files [PATTERN...]
         시스템에 설치된 유닛 파일 목록을 활성화 상태(is-enabled가 보고하는 대로)와 함께 나열합니다. 하나 이상의 PATTERN이 지정되면, 이름이 그중 하나와 일치하는 유닛 파일만 표시됩니다 (유닛 파일 시스템 경로와 일치하는 패턴은 지원되지 않습니다).

         list-units와 달리 이 명령어는 명시적으로 인스턴스화된 유닛 외에도 템플릿 유닛을 나열합니다.

         버전 233에 추가되었습니다.

     enable UNIT..., enable PATH...
         하나 이상의 유닛 또는 유닛 인스턴스를 활성화합니다. 이는 지정된 유닛 파일의 [Install] 섹션에 인코딩된 심볼릭 링크 세트를 생성합니다. 심볼릭 링크가 생성된 후, 변경 사항이 즉시 반영되도록 시스템 관리자 구성이 다시 로드됩니다 (daemon-reload와 동등한 방식으로). 이 작업이 활성화되는 유닛 중 어느 것도 시작하는 효과가 있다는 점에 유의하십시오. 이것이 원하는 경우, 이 명령어를 --now 스위치와 결합하거나 나중에 적절한 인수로 start를 호출하십시오. 유닛 인스턴스 활성화(예: foo@bar.service 형태의 유닛 활성화)의 경우, 인스턴스와 동일한 이름의 심볼릭 링크가 유닛 구성 디렉토리에 생성되지만, 이 링크들은 해당 인스턴스가 인스턴스화된 단일 템플릿 유닛 파일을 가리킵니다.

         이 명령어는 유효한 유닛 이름(이 경우 적절한 이름의 유닛 파일을 위해 다양한 유닛 파일 디렉토리가 자동으로 검색됨) 또는 유닛 파일의 절대 경로(이 경우 해당 파일이 직접 읽힘)를 예상합니다. 지정된 유닛 파일이 일반적인 유닛 파일 디렉토리 외부에 위치하는 경우, 유닛 구성 경로에 연결하는 추가 심볼릭 링크가 생성되어 start와 같은 명령어가 요청할 때 해당 파일이 발견되도록 보장합니다. 연결된 유닛 파일이 위치한 파일 시스템은 systemd가 시작될 때 접근 가능해야 합니다 (예: /home/ 또는 /var/ 아래의 모든 것은 루트 파일 시스템에 해당 디렉토리가 위치하지 않는 한 허용되지 않습니다).

         이 명령어는 실행된 파일 시스템 작업을 출력합니다. 이 출력은 --quiet을 전달하여 억제할 수 있습니다.

         이 작업은 유닛 파일의 [Install] 섹션에 제안된 심볼릭 링크만 생성한다는 점에 유의하십시오. 이 명령어가 유닛 구성 디렉토리를 조작하는 권장 방법이지만, 관리자는 이 디렉토리 아래에 심볼릭 링크를 배치하거나 이동하여 수동으로 추가 변경을 할 수 있습니다. 이는 제안된 기본 설치와 다른 구성을 생성하는 데 특히 유용합니다. 이 경우, 변경 사항이 반영되도록 daemon-reload를 수동으로 호출해야 합니다.

         설치 정보가 없는 유닛에 이 작업을 사용할 때 경고가 표시됩니다. --no-warn을 사용하여 경고를 억제할 수 있습니다.

         유닛 활성화는 start 명령어로 수행되는 유닛 시작(활성화)과 혼동해서는 안 됩니다. 유닛 활성화와 시작은 직교합니다. 유닛은 시작되지 않고 활성화될 수 있으며, 활성화되지 않고 시작될 수도 있습니다. 활성화는 단순히 유닛을 다양한 제안된 위치에 연결합니다 (예를 들어, 유닛이 부팅 시 또는 특정 유형의 하드웨어가 연결될 때 자동으로 시작되도록). 시작은 실제로 데몬 프로세스를 생성하거나 (서비스 유닛의 경우), 소켓을 바인딩하거나 (소켓 유닛의 경우) 등을 수행합니다.

         --system, --user, --runtime 또는 --global이 지정되었는지 여부에 따라, 이는 시스템용, 호출하는 사용자 전용, 시스템의 이번 부팅에만, 또는 모든 사용자의 향후 로그인에 대해 유닛을 활성화합니다. 마지막 경우, 시스템d 데몬 구성은 다시 로드되지 않는다는 점에 유의하십시오.

         마스크된 유닛에 enable을 사용하는 것은 지원되지 않으며 오류가 발생합니다.

     disable UNIT...
         하나 이상의 유닛을 비활성화합니다. 이는 지정된 유닛을 지원하는 유닛 파일에 대한 모든 심볼릭 링크를 유닛 구성 디렉토리에서 제거하며, 따라서 enable 또는 link로 이루어진 모든 변경 사항을 취소합니다. 이 작업은 enable 또는 link에 의해 실제로 생성된 심볼릭 링크뿐만 아니라 수동으로 생성된 심볼릭 링크를 포함하여 일치하는 유닛 파일에 대한 모든 심볼릭 링크를 제거한다는 점에 유의하십시오. disable은 enable의 효과를 취소하지만, 두 명령어는 그렇지 않다는 점에 유의하십시오. 왜냐하면 disable이 동일 유닛의 이전 enable 호출이 생성한 것보다 더 많은 심볼릭 링크를 제거할 수 있기 때문입니다.

         이 명령어는 유효한 유닛 이름만 예상하며, 유닛 파일 경로를 허용하지 않습니다.

         인수로 지정된 유닛 외에도, 작업 중인 유닛 파일의 [Install] 섹션에 포함된 Also= 설정에 나열된 모든 유닛이 비활성화됩니다.

         이 명령어는 작업을 완료한 후 시스템 관리자 구성을 암묵적으로 다시 로드합니다. 이 명령어는 비활성화되는 유닛을 암묵적으로 중지하지 않는다는 점에 유의하십시오. 이것이 원하는 경우, 이 명령어를 --now 스위치와 결합하거나 나중에 적절한 인수를 사용하여 stop 명령을 호출하십시오.

         이 명령어는 실행된 파일 시스템 작업(심볼릭 링크 제거)에 대한 정보를 출력합니다. 이 출력은 --quiet을 전달하여 억제할 수 있습니다.

         유닛이 비활성화되지만 트리거링 유닛이 여전히 활성 상태인 경우, 트리거링 유닛의 이름이 포함된 경고가 표시됩니다. --no-warn을 사용하여 경고를 억제할 수 있습니다.

         이 명령어를 --user와 함께 사용할 때, 작업 중인 유닛은 여전히 전역 범위에서 활성화되어 있을 수 있으며, 따라서 사용자 범위에서 성공적으로 비활성화된 후에도 자동으로 시작될 수 있습니다. 이 경우, --no-warn을 사용하여 억제할 수 있는 경고가 표시됩니다.

     mask UNIT...

     revert UNIT...
         하나 이상의 유닛 파일을 벤더 버전으로 되돌립니다. 이 명령어는 지정된 유닛을 수정하는 드롭인 구성 파일을 제거하며, 일치하는 벤더 제공 유닛 파일을 재정의하는 사용자 구성 유닛 파일도 제거합니다. 구체적으로, 유닛 "foo.service"의 경우, 영구 및 런타임 구성 디렉토리(즉, /etc/systemd/system 및 /run/systemd/system 아래)에 있는 모든 포함 파일을 가진 일치하는 디렉토리 "foo.service.d/"가 제거됩니다. 만약 유닛 파일에 벤더 제공 버전이 있는 경우(즉, /usr/ 아래에 위치한 유닛 파일), 이를 재정의하는 일치하는 영구 또는 런타임 유닛 파일도 제거됩니다. 유닛 파일에 벤더 제공 버전이 없는 경우(즉, /etc/systemd/system 또는 /run/systemd/system 아래에만 정의되어 있고 /usr/ 아래에 저장된 유닛 파일에는 없는 경우), 해당 파일은 제거되지 않는다는 점에 유의하십시오. 또한, 유닛이 마스킹된 경우 마스킹이 해제됩니다.

   Environment Commands
     systemd는 매니저가 생성하는 프로세스에 전달되는 환경 블록을 지원합니다. 변수 이름에는 ASCII 문자, 숫자, 밑줄 문자를 포함할 수 있습니다. 변수 이름은 비어 있거나 숫자로 시작할 수 없습니다. 변수 값에서는 대부분의 문자가 허용되지만, 전체 시퀀스는 유효한 UTF-8이어야 합니다. (줄 바꿈(NL), 탭(TAB), 또는 이스케이프 문자(ESC)와 같은 제어 문자는 유효한 ASCII이며 따라서 유효한 UTF-8이라는 점에 유의하십시오). 환경 블록의 전체 길이는 sysconf(3)에 정의된 _SC_ARG_MAX 값으로 제한됩니다.

     show-environment
         systemd 매니저 환경 블록을 출력합니다. 이는 매니저가 생성하는 모든 프로세스에 전달되는 환경 블록입니다. 환경 블록은 대부분의 셸에서 소싱하기에 적합한 단순한 형태로 출력됩니다. 변수 값에 특수 문자나 공백이 없는 경우 이스케이프 처리는 수행되지 않으며 할당은 "VARIABLE=value" 형태를 가집니다. 공백이나 셸에서 특별한 의미를 갖는 문자가 있는 경우 달러-단일-따옴표 이스케이프가 사용되며 할당은 "VARIABLE=$'value'" 형태를 가집니다. 이 구문은 bash(1), zsh(1), ksh(1), 그리고 busybox(1)의 ash(1)에서 지원되는 것으로 알려져 있지만 dash(1)나 fish(1)에서는 지원되지 않습니다.

         이것은 유효한 블록, 즉 구성 파일, 환경 생성기, 그리고 IPC(아래에 설명된 set-environment를 통해)를 통해 구성된 환경 변수의 조합을 보여준다는 점에 유의하십시오. 유닛 프로세스가 포크될 때, 이 결합된 환경 블록은 단위별 환경 변수와 추가로 결합되며, 이 변수들은 이 명령어에서는 보이지 않습니다.

     set-environment VARIABLE=VALUE...
         명령줄에 지정된 하나 이상의 서비스 매니저 환경 변수를 설정합니다. 이 명령어는 변수 이름과 값이 위에 나열된 규칙을 준수하지 않으면 실패합니다.

         이것이 서비스 매니저 구성 및 환경 생성기로부터 구성된 환경 블록과는 별개의 환경 블록에 작동한다는 점에 유의하십시오. 프로세스가 호출될 때마다 두 블록은 결합되고 (서비스별 환경 변수도 통합되며) 프로세스에 전달됩니다. show-environment 명령은 블록의 조합을 보여줍니다. 위를 참조하십시오.

         버전 233에 추가되었습니다.

     unset-environment VARIABLE...
         하나 이상의 systemd 매니저 환경 변수를 해제합니다. 변수 이름만 지정된 경우, 값에 관계없이 제거됩니다. 변수와 값이 지정된 경우, 해당 변수는 지정된 값과 일치할 때만 제거됩니다.

         이것이 서비스 매니저 구성 및 환경 생성기로부터 구성된 환경 블록과는 별개의 환경 블록에 작동한다는 점에 유의하십시오. 프로세스가 호출될 때마다 두 블록은 결합되고 (서비스별 환경 변수도 통합되며) 프로세스에 전달됩니다. show-environment 명령은 블록의 조합을 보여줍니다. 위를 참조하십시오. 이것은 이 명령을 서비스 매니저 구성 파일이나 생성기를 통해 정의된 환경 변수를 해제하는 데 사용할 수 없다는 것을 의미합니다.

         버전 233에 추가되었습니다.

     import-environment VARIABLE...
         클라이언트에서 설정된 모든 하나 이상의 환경 변수를 systemd 매니저 환경 블록으로 가져옵니다. 환경 변수 이름 목록이 전달되면, 클라이언트 측 값은 매니저의 환경 블록으로 가져와집니다. 위에 설명된 규칙에 따라 유효한 환경 변수 이름이 아니거나 유효하지 않은 값을 가진 이름이 있으면 오류가 발생합니다. 인수가 전달되지 않으면 systemctl 프로세스가 상속받은 전체 환경 블록이 가져와집니다. 이 모드에서는 상속된 유효하지 않은 환경 변수는 조용히 무시됩니다.

         전체 상속된 환경 블록을 가져오는 것(인수 없이 이 명령을 호출하는 것)은 사용 중단되었습니다. 셸은 로컬에서만 의미가 있고 셸의 자손 프로세스만을 위한 수십 개의 변수를 설정합니다. 이러한 변수들이 전역 환경 블록에 있으면 다른 프로세스에게 혼란을 줍니다.

         버전 209에 추가되었습니다.

   Manager State Commands
     daemon-reload
         systemd 매니저 구성을 다시 로드합니다. 이는 모든 생성기(systemd.generator(7) 참조)를 다시 실행하고, 모든 유닛 파일을 다시 로드하며, 전체 종속성 트리를 재구성합니다. 데몬이 다시 로드되는 동안, systemd가 사용자 구성을 대신하여 수신하는 모든 소켓은 계속 액세스 가능합니다.

         이 명령어는 reload 명령어와 혼동해서는 안 됩니다.

     daemon-reexec
         systemd 매니저를 재실행합니다. 이는 매니저 상태를 직렬화하고, 프로세스를 재실행한 다음, 상태를 다시 역직렬화합니다. 이 명령어는 디버깅 및 패키지 업그레이드를 제외하고는 거의 사용되지 않습니다. 때로는 무거운 daemon-reload로 유용할 수 있습니다. 데몬이 재실행되는 동안, systemd가 사용자 구성을 대신하여 수신하는 모든 소켓은 계속 액세스 가능합니다.

     log-level [LEVEL]
         인수가 제공되지 않으면 매니저의 현재 로그 레벨을 출력합니다. 선택적 인수 LEVEL이 제공되면, 이 명령어는 매니저의 현재 로그 레벨을 LEVEL로 변경합니다 (systemd(1)에 설명된 --log-level=과 동일한 값을 허용합니다).

     service-watchdogs [yes|no]

     default

     rescue
         복구 모드로 진입합니다. 이는 systemctl isolate rescue.target과 동일합니다. 이 작업은 기본적으로 차단되며, 비동기 동작을 요청하려면 --no-block을 사용하십시오.

     halt
         시스템을 종료하고 정지시킵니다. 이는 대부분 systemctl start halt.target --job-mode=replace-irreversibly --no-block과 동일하지만, 모든 사용자에게 wall 메시지를 출력합니다. 이 명령은 비동기적입니다. 정지 작업이 큐에 추가된 후 반환되며, 완료될 때까지 기다리지 않습니다. 이 작업은 시스템을 종료한 후 OS 커널을 단순히 정지시키고 하드웨어는 전원이 켜진 상태로 남겨둔다는 점에 유의하십시오. 시스템 전원을 끄려면 systemctl poweroff를 사용하십시오(아래 참조).

         --force와 결합되면 모든 실행 중인 서비스의 종료가 건너뛰어지지만, 모든 프로세스는 종료되고 모든 파일 시스템은 마운트 해제되거나 읽기 전용으로 마운트된 후 즉시 시스템이 정지됩니다. --force가 두 번 지정되면, 어떤 프로세스도 종료하거나 파일 시스템을 마운트 해제하지 않고 즉시 작업이 실행됩니다. 이로 인해 데이터 손실이 발생할 수 있습니다. --force가 두 번 지정될 때 정지 작업은 systemctl 자체에 의해 실행되며 시스템 관리자에게 연락하지 않습니다. 이는 시스템 관리자가 충돌했더라도 명령이 성공해야 함을 의미합니다.

         --when=과 결합되면 종료가 주어진 타임스탬프 이후에 예약됩니다. 그리고 --when=cancel은 종료를 취소합니다.

     poweroff
         시스템을 종료하고 전원을 끕니다. 이는 대부분 systemctl start poweroff.target --job-mode=replace-irreversibly --no-block과 동일하지만, 모든 사용자에게 wall 메시지를 출력합니다. 이 명령은 비동기적입니다. 전원 끄기 작업이 큐에 추가된 후 반환되며, 완료될 때까지 기다리지 않습니다.

         이 명령은 halt와 유사한 방식으로 --force와 --when=을 처리합니다.

     reboot
         시스템을 종료하고 재부팅합니다.

         이 명령은 대부분 systemctl start reboot.target --job-mode=replace-irreversibly --no-block과 동일하지만, 모든 사용자에게 wall 메시지를 출력합니다. 이 명령은 비동기적입니다. 재부팅 작업이 큐에 추가된 후 반환되며, 완료될 때까지 기다리지 않습니다.

         --reboot-argument= 스위치를 제공하면, 이는 reboot(2) 시스템 호출의 선택적 인수로 전달됩니다.

         이 명령은 halt와 유사한 방식으로 --force와 --when=을 처리합니다.

     kexec
         kexec를 통해 시스템을 종료하고 재부팅합니다. 이 명령은 아직 로드되지 않은 kexec 커널을 로드하거나 실패합니다. 커널은 별도의 단계에서 더 일찍 로드될 수 있으며, 이는 사용자 정의 initrd 또는 추가 커널 명령줄 옵션이 필요한 경우 특히 유용합니다. --force는 kexec 커널 없이 계속 진행하여 일반 재부팅을 수행하는 데 사용될 수 있습니다. 최종 재부팅 단계는 systemctl start kexec.target --job-mode=replace-irreversibly --no-block과 동일합니다.

         커널을 로드하려면 UAPI.1 Boot Loader Specification[1]을 따라 열거가 수행되며 기본 부팅 항목이 로드됩니다. 이 단계가 성공하려면 시스템이 UEFI를 사용해야 하며 부트로더 항목이 적절하게 구성되어야 합니다. bootctl list를 사용하여 부팅 항목을 나열할 수 있습니다. bootctl(1)을 참조하십시오.

         이 명령은 비동기적입니다. 재부팅 작업이 큐에 추가된 후 반환되며, 완료될 때까지 기다리지 않습니다.

         이 명령은 halt와 유사하게 --force와 --when=을 처리합니다.

         kexec --load를 통해 새 커널이 로드된 경우, "SYSTEMCTL_SKIP_AUTO_KEXEC=1"이 설정되지 않는 한 재부팅이 호출될 때 kexec가 수행됩니다.

     soft-reboot
         사용자 공간을 종료하고 재부팅합니다. 이는 systemctl start
         soft-reboot.target --job-mode=replace-irreversibly --no-block과 동일합니다. 이
         명령어는 비동기적입니다. 재부팅 작업이 큐에 추가된 후 반환되며, 완료될 때까지 기다리지 않습니다.

         이 명령어는 halt와 유사한 방식으로 --force 및 --when=을 준수합니다.

         이 작업은 커널을 실행 상태로 유지한 채 사용자 공간만 재부팅합니다. 자세한 내용은
         systemd-soft-reboot.service(8)를 참조하십시오.

         "/run/nextroot/"에 새로운 루트 파일 시스템이 설정된 경우, "SYSTEM‐
         CTL_SKIP_AUTO_SOFT_REBOOT=1"이 설정되지 않는 한 재부팅이 호출될 때 soft-reboot이 수행됩니다.

         버전 254에 추가되었습니다.

     exit [EXIT_CODE]
         서비스 관리자에게 종료를 요청합니다. 이는 사용자 서비스 관리자(즉, --user 옵션과 함께 사용될 때) 또는 컨테이너에서만 지원되며, 그렇지 않은 경우에는 poweroff와 동일합니다. 이 명령어는 비동기적입니다. 종료 작업이 큐에 추가된 후 반환되며, 완료될 때까지 기다리지 않습니다.

         EXIT_CODE가 전달되면 서비스 관리자는 지정된 종료 코드로 종료됩니다.

         버전 227에 추가되었습니다.

     switch-root [ROOT [INIT]]
         다른 루트 디렉터리로 전환하고 그 아래에 새로운 시스템 관리자 프로세스를 실행합니다. 이는 initrd에서 사용하도록 의도되었으며, initrd의 시스템 관리자 프로세스(별칭 "init" 프로세스, PID 1)에서 실제 호스트 루트 파일 시스템에서 로드된 메인 시스템 관리자 프로세스로 전환합니다. 이 호출은 두 개의 인수를 받습니다. 새로운 루트 디렉터리가 될 디렉터리와 PID 1로 실행될 새로운 시스템 관리자 바이너리의 경로입니다. 둘 다 생략되거나 전자가 빈 문자열인 경우 /sysroot/로 기본 설정됩니다. 후자가 생략되거나 빈 문자열인 경우, systemd 바이너리가 자동으로 검색되어 서비스 관리자로 사용됩니다. 시스템 관리자 경로가 생략되거나 빈 문자열과 같거나 systemd 바이너리 경로와 동일한 경우, initrd의 시스템 관리자 프로세스 상태가 메인 시스템 관리자에게 전달되어 initrd 부트 단계에 관련된 서비스 상태를 나중에 검사할 수 있게 합니다.

         버전 209에 추가되었습니다.

     sleep
         시스템을 절전, 최대 절전 모드, 하이브리드 절전 모드 또는 절전 후 최대 절전 모드로 전환합니다. 사용할 절전 작업은 systemd-logind.service(8)에 의해 자동으로 선택됩니다. 기본적으로 suspend-then-hibernate가 사용되며, 지원되지 않는 경우 suspend로 폴백하고 그 다음 hibernate합니다. 자세한 내용은 logind.conf(5)의 SleepOperation= 설정을 참조하십시오. 이 명령어는 비동기적이며, 절전 작업이 성공적으로 큐에 추가된 후 반환됩니다. 절전/재개 주기가 완료될 때까지 기다리지 않습니다.

         버전 256에 추가되었습니다.

     suspend
         시스템을 절전 상태로 전환합니다. 이는 특별한 타겟 유닛 suspend.target의 활성화를 유발합니다. 이 명령어는 비동기적이며, 절전 작업이 성공적으로 큐에 추가된 후 반환됩니다. 절전/재개 주기가 완료될 때까지 기다리지 않습니다.

         --force가 지정되었고 systemd-logind가 해당 작업에 대해 오류를 반환한 경우, 오류는 무시되고 타겟 유닛을 직접 시작하여 작업이 다시 시도됩니다.

     hibernate
         시스템을 최대 절전 모드로 전환합니다. 이는 특별한 타겟 유닛 hibernate.target의 활성화를 유발합니다. 이 명령어는 비동기적이며, 최대 절전 모드 작업이 성공적으로 큐에 추가된 후 반환됩니다. 최대 절전 모드/복원 주기가 완료될 때까지 기다리지 않습니다.

         이 명령어는 suspend와 동일한 방식으로 --force를 준수합니다.

     hybrid-sleep
         시스템을 최대 절전 모드 및 절전 상태로 전환합니다. 이는 특별한 타겟 유닛 hybrid-sleep.target의 활성화를 유발합니다. 이 명령어는 비동기적이며, 하이브리드 절전 작업이 성공적으로 큐에 추가된 후 반환됩니다. 절전/깨우기 주기가 완료될 때까지 기다리지 않습니다.

         이 명령어는 suspend와 동일한 방식으로 --force를 준수합니다.

         버전 196에 추가되었습니다.

     suspend-then-hibernate
         배터리가 부족하거나 systemd-sleep.conf에 지정된 지연 시간이 경과했을 때 시스템을 절전 상태로 전환하고 최대 절전 모드로 전환합니다. 이는 특별한 타겟 유닛 suspend-then-hibernate.target의 활성화를 유발합니다. 이 명령어는 비동기적이며, 하이브리드 절전 작업이 성공적으로 큐에 추가된 후 반환됩니다. 절전/깨우기 또는 최대 절전 모드/복원 주기가 완료될 때까지 기다리지 않습니다.

         이 명령어는 suspend와 동일한 방식으로 --force를 준수합니다.

         버전 240에 추가되었습니다.

   Parameter Syntax
     위에 나열된 유닛 명령어는 단일 유닛 이름(UNIT으로 지정됨) 또는 여러 유닛 사양(PATTERN...으로 지정됨)을 사용합니다. 첫 번째 경우, 접미사가 있든 없든 유닛 이름을 지정해야 합니다. 접미사가 지정되지 않은 경우(유닛 이름이 "약어"인 경우), systemctl은 적절한 접미사(기본적으로 ".service")를 추가하며, 특정 유닛 유형에만 작동하는 명령어의 경우 유형별 접미사를 추가합니다. 예를 들어,

         # systemctl start sshd

     와

         # systemctl start sshd.service

     는 동일하며,

         # systemctl isolate default

     와

         # systemctl isolate default.target

     도 동일합니다.

     장치 노드에 대한 (절대) 경로는 자동으로 장치 유닛 이름으로 변환되며, 마운트 유닛 이름에 대한 다른 (절대) 경로는 변환됩니다.

         # systemctl status /dev/sda
         # systemctl status /home

     는 다음과 동일합니다.

         # systemctl status dev-sda.device
         # systemctl status home.mount

     두 번째 경우, 셸 스타일의 글롭은 현재 메모리에 있는 모든 유닛의 기본 이름과 일치합니다. 접미사가 있든 없든 리터럴 유닛 이름은 첫 번째 경우와 동일하게 처리됩니다. 이는 리터럴 유닛 이름이 항상 정확히 하나의 유닛을 참조한다는 것을 의미하지만, 글롭은 0개의 유닛과 일치할 수 있으며 이는 오류로 간주되지 않습니다.

     글롭 패턴은 fnmatch(3)을 사용하므로 일반적인 셸 스타일 글롭 규칙이 사용되며, "*" , "?" , "[]"를 사용할 수 있습니다. 자세한 내용은 glob(7)을 참조하십시오. 패턴은 현재 메모리에 있는 유닛의 기본 이름과 일치하며, 아무것도 일치하지 않는 패턴은 조용히 건너뜁니다. 예를 들어,

         # systemctl stop "sshd@*.service"

     는 모든 sshd@.service 인스턴스를 중지합니다. 유닛의 별칭 이름과 메모리에 없는 유닛은 글롭 확장에 고려되지 않는다는 점에 유의하십시오.

     유닛 파일 명령어의 경우, 지정된 UNIT은 유닛 파일의 이름(가능하게는 약어, 위 참조) 또는 유닛 파일의 절대 경로여야 합니다.

         # systemctl enable foo.service

     또는

         # systemctl link /path/to/foo.service

OPTIONS
     다음 옵션들이 이해됩니다:

     -t, --type=
         인수는 service나 socket과 같은 단위 유형의 쉼표로 구분된 목록입니다. list-units, list-dependencies,
         show 또는 status를 사용하여 단위를 나열할 때, 지정된 유형의 단위만 표시됩니다. 기본적으로
         모든 유형의 단위가 표시됩니다.

         특별한 경우로, 인수 중 하나가 help인 경우 허용되는 값 목록이 출력되고 프로그램이 종료됩니다.

     --state=
         인수는 단위의 LOAD, SUB 또는 ACTIVE 상태를 쉼표로 구분하여 나열합니다. list-units, list-dependencies, show 또는
         status를 사용하여 단위를 나열할 때, 지정된 상태에 있는 단위만 표시됩니다. --state=failed 또는
         --failed를 사용하여 실패한 단위만 표시할 수 있습니다.

         특별한 경우로, 인수 중 하나가 help인 경우 허용되는 값 목록이 출력되고 프로그램이 종료됩니다.

         버전 206에 추가되었습니다.

     -p, --property=
         show 명령을 사용하여 단위/작업/관리자 속성을 표시할 때, 인수에 지정된 속성으로 표시를 제한합니다. 인수는 "MainPID"와 같은 속성 이름의 쉼표로 구분된 목록이어야 합니다. 지정하지 않으면 알려진 모든 속성이 표시됩니다. 한 번 이상 지정하면 지정된 이름의 모든 속성이 표시됩니다. 속성 이름에 대해 셸 자동 완성이 구현되어 있습니다.

         관리자 자체의 경우, systemctl show는 사용 가능한 모든 속성을 표시하며, 이 중 대부분은 systemd-system.conf(5)에 설명된 옵션에서 파생되거나 밀접하게 일치합니다.

         단위에 대한 속성은 단위 유형에 따라 다르므로, 어떤 단위(존재하지 않는 단위라도)를 표시하는 것은 해당 유형에 속하는 속성을 나열하는 방법입니다. 마찬가지로, 어떤 작업을 표시하면 모든 작업에 속하는 속성이 나열됩니다. 단위에 대한 속성은 systemd.unit(5)에 문서화되어 있으며, 개별 단위 유형에 대한 페이지는 systemd.service(5), systemd.socket(5) 등입니다.

     -P
         --value --property=와 동일하며, 즉 속성 이름이나 "=" 없이 속성 값만 표시합니다. -P를 한 번 사용하는 경우 -p/--property=로 나열된 모든 속성에 영향을 미친다는 점에 유의하십시오.

         버전 246에 추가되었습니다.

     -a, --all
         list-units를 사용하여 단위를 나열할 때, 비활성 단위와 다른 단위를 따르는 단위도 표시합니다. 단위/작업/관리자 속성을 표시할 때, 설정되었는지 여부와 관계없이 모든 속성을 표시합니다.

         파일 시스템에 설치된 모든 단위를 나열하려면 list-unit-files 명령을 사용하십시오.

         list-dependencies를 사용하여 단위를 나열할 때, 모든 종속 단위의 종속성을 재귀적으로 표시합니다 (기본적으로 대상 단위의 종속성만 표시됩니다).

         status와 함께 사용하면, 인쇄할 수 없는 문자를 포함하거나 매우 긴 경우에도 저널 메시지를 전체적으로 표시합니다. 기본적으로 인쇄할 수 없는 문자가 포함된 필드는 "blob data"로 축약됩니다. (페이지 매거인이 인쇄할 수 없는 문자를 다시 이스케이프 처리할 수 있다는 점에 유의하십시오.)

     -r, --recursive
         단위를 나열할 때, 로컬 컨테이너의 단위도 표시합니다. 로컬 컨테이너의 단위는 단일 콜론 문자(":")로 구분된 컨테이너 이름으로 접두사가 붙습니다.

         버전 212에 추가되었습니다.

     --reverse
         list-dependencies를 사용하여 단위 간의 역 종속성을 표시합니다. 즉, Wants= 및 유사한 대신 WantedBy=, RequiredBy=, UpheldBy=, PartOf=, BoundBy= 유형의 종속성을 따릅니다.

         버전 203에 추가되었습니다.

     --after
         list-dependencies와 함께 사용하면, 지정된 단위보다 먼저 순서가 지정된 단위를 표시합니다. 즉, After= 종속성을 따르는 단위를 재귀적으로 나열합니다.

         모든 After= 종속성은 Before= 종속성을 생성하도록 자동으로 미러링된다는 점에 유의하십시오. 시간적 종속성은 명시적으로 지정할 수 있지만, WantedBy= 대상 단위(systemd.target(5) 참조) 및 다른 지시문(예: RequiresMountsFor=)의 결과로 암묵적으로 생성되기도 합니다. 명시적으로 그리고 암묵적으로 도입된 종속성은 list-dependencies로 표시됩니다.

         list-jobs 명령에 전달할 때, 인쇄된 각 작업에 대해 어떤 다른 작업이 해당 작업을 기다리고 있는지 표시합니다. --before와 결합하여 각 작업을 기다리는 작업과 각 작업이 기다리는 모든 작업을 모두 표시할 수 있습니다.

         버전 203에 추가되었습니다.

     --before
         list-dependencies와 함께 사용하면, 지정된 단위보다 나중에 순서가 지정된 단위를 표시합니다. 즉, Before= 종속성을 따르는 단위를 재귀적으로 나열합니다.

     --job-mode=
         새로운 작업을 큐에 넣을 때, 이 옵션은 이미 큐에 있는 작업을 어떻게 처리할지 제어합니다. 이 옵션은 "fail", "lenient", "replace", "re‐place-irreversibly", "isolate", "ignore-dependencies", "ignore-requirements", "flush", "triggering", 또는 "restart-dependencies" 중 하나를 받습니다. "isolate" 명령이 사용되는 경우를 제외하고는 "replace"로 기본 설정됩니다.

         "fail"이 지정되고 약한 종속성에 대한 요청된 작업이 보류 중인 작업과 충돌하는 경우(더 구체적으로는 이미 보류 중인 시작 작업이 중지 작업으로 되돌아가거나 그 반대의 경우), 해당 작업을 실패시킵니다.

         "replace-irreversibly"가 지정되면, "replace"처럼 작동하지만 새로운 작업들을 되돌릴 수 없는 것으로 표시합니다. 이는 향후 충돌하는 트랜잭션이 이 작업들을 대체하는 것을 방지합니다(또는 되돌릴 수 없는 작업들이 아직 보류 중인 동안 큐에 추가되는 것조차 방지합니다). 되돌릴 수 없는 작업은 여전히 cancel 명령을 사용하여 취소할 수 있습니다. 이 작업 모드는 shutdown.target을 포함하는 모든 트랜잭션에 사용되어야 합니다.

         "isolate"는 시작 작업에만 유효하며, 지정된 유닛이 시작될 때 다른 모든 유닛을 중지시킵니다. 이 모드는 isolate 명령이 사용될 때 항상 사용됩니다.

         "flush"는 새 작업이 큐에 추가될 때 모든 큐에 있는 작업을 취소시킵니다.

         "ignore-dependencies"가 지정되면, 이 새 작업에 대해 모든 유닛 종속성이 무시되며 작업이 즉시 실행됩니다. 이 옵션을 전달하면 전달된 유닛의 필요한 유닛은 끌어와지지 않으며, 순서 종속성도 존중되지 않습니다. 이는 주로 관리자를 위한 디버깅 및 복구 도구이며 애플리케이션에서 사용해서는 안 됩니다.

         "ignore-requirements"는 "ignore-dependencies"와 유사하지만, 요구 사항 종속성만 무시하며 순서 종속성은 여전히 존중됩니다.

         "triggering"은 systemctl stop과 함께만 사용될 수 있습니다. 이 모드에서 지정된 유닛과 이를 트리거하는 모든 활성 유닛이 중지됩니다. 트리거링 유닛에 대한 자세한 내용은 systemd.unit(5)의 Triggers= 논의를 참조하십시오.

         "restart-dependencies"는 systemctl start와 함께만 사용될 수 있습니다. 이 모드에서 지정된 유닛의 종속성은 해당 유닛에 대해 재시작 작업이 큐에 추가된 것처럼 재시작 전파를 받습니다.

     --kill-whom=
         kill과 함께 사용될 때, UNIX 프로세스 신호를 보낼 프로세스를 선택합니다. main, control, cgroup 또는 all 중 하나여야 하며, 이는 main 프로세스만, control 프로세스, 유닛의 제어 그룹 내 모든 프로세스 또는 유닛의 모든 프로세스를 종료할지 선택합니다. 유닛의 main 프로세스는 유닛의 수명을 정의하는 프로세스입니다. 유닛의 control 프로세스는 관리자에 의해 호출되어 유닛의 상태 변경을 유도하는 프로세스입니다. 예를 들어, 서비스 유닛의 ExecStartPre=, ExecStop= 또는 ExecReload= 설정으로 인해 시작된 모든 프로세스는 control 프로세스입니다. 한 번에 하나의 상태 변경만 실행되므로, 유닛당 한 번에 하나의 control 프로세스만 있다는 점에 유의하십시오. Type=forking 서비스의 경우, 관리자가 ExecStart=를 위해 시작하는 초기 프로세스는 control 프로세스이며, 궁극적으로 이 프로세스에 의해 포크된 프로세스는 유닛의 main 프로세스로 간주됩니다(확인 가능한 경우). 이는 다른 유형의 서비스 유닛과는 다릅니다. 다른 유형의 서비스 유닛의 경우, 관리자가 ExecStart=를 위해 포크한 프로세스는 항상 main 프로세스 자체입니다. 서비스 유닛은 0개 또는 1개의 main 프로세스, 0개 또는 1개의 control 프로세스 및 유닛의 제어 그룹에 속하는 추가 프로세스 수로 구성됩니다. 그러나 모든 유닛 유형이 이러한 유형의 프로세스를 관리하는 것은 아닙니다. 예를 들어, mount 유닛의 경우, control 프로세스가 정의되지만(이는 /usr/bin/mount 및 /usr/bin/umount의 호출입니다), main 프로세스는 정의되지 않습니다. 생략하면 all로 기본 설정되지만, --kill-subgroup=이 사용되는 경우에는 cgroup으로 기본 설정됩니다.

     --kill-subgroup=PATH
         kill 명령과 함께 사용하기 위해 신호를 보낼 제어 그룹 하위 경로를 지정합니다. 기본적으로 선택된 신호는 유닛의 cgroup의 모든 프로세스(main/control 프로세스도 포함(외부에 있는 경우) – --kill-whom=의 적용을 받음)에 전달됩니다. 그러나 이 옵션을 사용하면 대신 하위 그룹을 선택할 수 있습니다. 이 기능은 "cgroup" 또는 "cgroup-fail"이 --kill-whom=과 함께 사용될 때만 사용 가능하며, 실제로 --kill-subgroup=이 사용될 경우 전자가 기본값입니다.

         지정된 경로는 슬래시로 접두사 지정될 수도 있고 아닐 수도 있으며, 그 유무는 영향을 미치지 않습니다. 해당 경로는 유닛의 main 제어 그룹 경로를 기준으로 상대적으로 취해집니다.

         이 기능은 제어 그룹 위임이 활성화된 유닛에서만 사용 가능합니다(systemd.resource-control(5)의 Delegate= 참조).

     -s, --signal=
         kill과 함께 사용될 때, 선택된 프로세스에 보낼 신호를 선택합니다. SIGTERM, SIGINT 또는 SIGSTOP과 같은 잘 알려진 신호 지정자 중 하나여야 합니다. 생략하면 SIGTERM으로 기본 설정됩니다.

         특별한 값 "help"는 알려진 값들을 나열하고 프로그램은 즉시 종료되며, 특별한 값 "list"는 알려진 값들과 수치 신호 번호를 나열하고 프로그램은 즉시 종료됩니다.

     --what=
         clean 명령이 호출될 때 제거할 유닛별 리소스 유형을 선택합니다. 위를 참조하십시오. 리소스 유형을 선택하기 위해 configuration, state, cache, logs, runtime, fdstore 중 하나를 지정합니다. 이 옵션은 여러 번 지정될 수 있으며, 이 경우 지정된 모든 리소스 유형이 제거됩니다. 또한 모든 여섯 가지 리소스 유형을 지정하는 단축키로 특별한 값 all을 허용합니다. 이 옵션을 지정하지 않으면 cache, runtime 및 fdstore의 조합으로 기본 설정되며, 이는 일반적으로 중복으로 간주되어 다음 호출에서 재구성될 수 있는 세 가지 종류의 리소스입니다. fdstore 리소스 유형의 명시적 제거는 FileDescriptorStorePreserve= 옵션이 활성화된 경우에만 유용하다는 점에 유의하십시오. 그렇지 않으면 유닛이 중지될 때 파일 디스크립터 저장소가 자동으로 정리됩니다.

         버전 243에 추가되었습니다.

     -f, --force
         enable과 함께 사용될 때, 기존의 충돌하는 심볼릭 링크를 덮어씁니다.

         edit과 함께 사용될 때, 이미 존재하지 않는 지정된 모든 유닛을 생성합니다.

         suspend, hibernate, hybrid-sleep 또는 suspend-then-hibernate와 함께 사용될 때, systemd-logind가 반환하는 오류는 무시되며, 해당 유닛을 시작함으로써 작업이 직접 수행됩니다.

         halt, poweroff, reboot 또는 kexec와 함께 사용될 때, 모든 유닛을 종료하지 않고 선택된 작업을 실행합니다. 그러나 모든 프로세스는 강제로 종료되고 모든 파일 시스템은 마운트 해제되거나 읽기 전용으로 다시 마운트됩니다. 따라서 이는 즉각적인 재부팅을 요청하기 위한 급진적이지만 비교적 안전한 옵션입니다. 이러한 작업에 대해 --force가 두 번 지정되면(kexec 제외), 프로세스를 종료하거나 파일 시스템을 마운트 해제하지 않고 즉시 실행됩니다.

             경고
             이러한 작업 중 어느 것과도 --force를 두 번 지정하면 데이터 손실이 발생할 수 있습니다. --force가 두 번 지정될 때 선택된 작업은 systemctl 자체에 의해 실행되며 시스템 관리자에게 연락하지 않는다는 점에 유의하십시오. 이는 시스템 관리자가 충돌했더라도 명령이 성공해야 함을 의미합니다.

     --message=
         halt, poweroff 또는 reboot과 함께 사용될 때, 작업 이유를 설명하는 짧은 메시지를 설정합니다. 이 메시지는 기본 종료 메시지와 함께 기록됩니다.

         버전 225에 추가되었습니다.

     --now
         enable, disable, mask 또는 reenable과 함께 사용될 때, 지정된 유닛 파일 작업이 성공한 후 유닛을 시작/중지/재시작합니다.

         버전 220에 추가되었습니다.

     --root=
         enable/disable/is-enabled(및 관련 명령)과 함께 사용될 때, 유닛 파일을 찾을 때 지정된 루트 경로를 사용합니다. 이 옵션이 있으면 systemctl은 변경 사항을 수행하기 위해 systemd 데몬과 통신하는 대신 파일 시스템에서 직접 작동합니다.

     --image=image
         디스크 이미지 파일 또는 블록 장치 노드의 경로를 지정합니다. 지정되면 모든 작업은 지정된 디스크 이미지의 파일 시스템에 적용됩니다. 이 옵션은 --root=와 유사하지만 디스크 이미지 또는 블록 장치에 저장된 파일 시스템에 대해 작동합니다. 디스크 이미지는 UAPI.2 Discoverable Partitions Specification[2]을 따르는 GPT 파티션 테이블 내에 파일 시스템만 포함하거나 일련의 파일 시스템을 포함해야 합니다. 지원되는 디스크 이미지에 대한 자세한 내용은 systemd-nspawn(1)의 동일한 이름의 switch를 참조하십시오.

         버전 252에 추가되었습니다.

     --image-policy=policy
         시스템d.image-policy(7)에 따라 이미지 정책 문자열을 인수로 받습니다. 이 정책은 --image=를 통해 지정된 디스크 이미지에서 작동할 때 적용됩니다. 지정하지 않으면 "*" 정책으로 기본 설정되며, 즉 이미지 내의 모든 인식된 파일 시스템이 사용됩니다.

     --runtime
         enable, disable, edit (및 관련 명령어)와 함께 사용될 때, 변경 사항은 일시적으로만 적용되어 다음 재부팅 시 손실됩니다. 이로 인해 변경 사항은 /etc/의 하위 디렉토리가 아닌 /run/에 이루어지지만, 즉각적인 효과는 동일합니다. 그러나 후자는 재부팅 시 손실되므로 변경 사항도 손실됩니다.

         마찬가지로, set-property와 함께 사용될 때도 변경 사항은 일시적으로만 적용되어 다음 재부팅 시 손실됩니다.

     --preset-mode=
         "full"(기본값), "enable-only", "disable-only" 중 하나를 받습니다. preset 또는 preset-all 명령어와 함께 사용될 때, 유닛이 preset 규칙에 따라 비활성화 및 활성화되어야 하는지, 아니면 활성화만 되어야 하는지, 또는 비활성화만 되어야 하는지를 제어합니다.

         버전 215에 추가되었습니다.

     -n, --lines=
         status와 함께 사용될 때, 가장 최근 항목부터 표시할 저널 라인 수를 제어합니다. 양의 정수 인수를 받거나 0을 입력하여 저널 출력을 비활성화할 수 있습니다. 기본값은 10입니다.

     -o, --output=
         status와 함께 사용될 때, 표시되는 저널 항목의 형식을 제어합니다. 사용 가능한 선택 사항은 journalctl(1)을 참조하십시오. 기본값은 "short"입니다.

     --firmware-setup
         reboot, poweroff 또는 halt 명령어와 함께 사용될 때, 시스템의 펌웨어에 다음 부팅 시 펌웨어 설정 인터페이스로 재부팅하도록 지시합니다. 이 기능이 모든 시스템에서 사용 가능한 것은 아님에 유의하십시오.

         버전 220에 추가되었습니다.

     --boot-loader-menu=timeout
         reboot, poweroff 또는 halt 명령어와 함께 사용될 때, 시스템의 부트 로더에 다음 부팅 시 부트 로더 메뉴를 표시하도록 지시합니다. 메뉴 시간 초과를 나타내는 시간 값을 매개변수로 받습니다. 메뉴 시간 초과를 비활성화하려면 0을 전달하십시오. 모든 부트 로더가 이 기능을 지원하는 것은 아님에 유의하십시오.

         버전 242에 추가되었습니다.

     --boot-loader-entry=ID
         reboot, poweroff 또는 halt 명령어와 함께 사용될 때, 시스템의 부트 로더에 다음 부팅 시 특정 부트 로더 항목으로 부팅하도록 지시합니다. 부트 로더 항목 식별자를 인수로 받거나 사용 가능한 항목을 나열하려면 "help"를 사용하십시오. 모든 부트 로더가 이 기능을 지원하는 것은 아님에 유의하십시오.

         버전 242에 추가되었습니다.

     --reboot-argument=
         이 스위치는 reboot와 함께 사용됩니다. 값은 아키텍처 및 펌웨어에 따라 다릅니다. 예를 들어, "recovery"는 시스템 복구를 트리거하는 데 사용될 수 있으며, "fota"는 "OTA(Over The Air) 펌웨어" 업데이트를 트리거하는 데 사용될 수 있습니다.

         버전 246에 추가되었습니다.

     --plain

         버전 203에 추가되었습니다.

     --timestamp=

         pretty (이것이 기본값입니다)
             "Day YYYY-MM-DD HH:MM:SS TZ"

             버전 248에 추가되었습니다.

         unix
             "@seconds-since-the-epoch"

             버전 251에 추가되었습니다.

         us, μs
             "Day YYYY-MM-DD HH:MM:SS.UUUUUU TZ"

             버전 248에 추가되었습니다.

         utc
             "Day YYYY-MM-DD HH:MM:SS UTC"

             버전 248에 추가되었습니다.

         us+utc, μs+utc
             "Day YYYY-MM-DD HH:MM:SS.UUUUUU UTC"

             버전 248에 추가되었습니다.

     --mkdir

         버전 248에 추가되었습니다.

     --drop-in=NAME
         edit와 함께 사용될 때, override.conf 대신 NAME을 드롭인 파일 이름으로 사용합니다.

         버전 253에 추가되었습니다.

     --when=
         halt, poweroff, reboot 또는 kexec와 함께 사용될 때, 작업을 주어진 타임스탬프에 수행하도록 예약합니다. 이 타임스탬프는 systemd.time(7) 섹션 "PARSING TIMESTAMPS"에 문서화된 구문을 준수해야 합니다. 특히, "show"가 주어지면 현재 예약된 작업이 표시되며, 빈 문자열이나 "cancel"을 전달하여 취소할 수 있습니다. "auto"는 유지보수 기간 또는 1분 후를 기준으로 작업을 예약합니다.

         버전 254에 추가되었습니다.

     --stdin
         edit와 함께 사용될 때, 파일의 내용은 표준 입력에서 읽어오며 편집기가 실행되지 않습니다. 이 모드에서는 파일의 기존 내용이 완전히 대체됩니다. 이는 스크립트에서 유닛 파일을 "편집"하는 데 유용합니다.

             $ systemctl edit --drop-in=limits.conf --stdin some-service.service <<EOF
             [Unit]
             AllowedCPUs=7,11
             EOF

         이 모드에서는 여러 드롭인 파일을 "편집"할 수 있으며, 동일한 내용이 모두에 작성됩니다.

         버전 256에 추가되었습니다.

     -H, --host=
         작업을 원격으로 실행합니다. 연결할 호스트 이름 또는 "@"로 구분된 사용자 이름 및 호스트 이름을 지정하십시오. 호스트 이름은 선택적으로 SSH가 수신 대기하는 포트를 ":"로 구분하여 접미사로 지정할 수 있으며, 그 후 "/"로 구분된 컨테이너 이름을 지정하여 지정된 호스트의 특정 컨테이너에 직접 연결할 수 있습니다. 이는 SSH를 사용하여 원격 머신 관리 인스턴스와 통신합니다. 컨테이너 이름은 machinectl -H HOST로 나열할 수 있습니다. IPv6 주소는 대괄호 안에 넣으십시오.

     -M, --machine=
         로컬 컨테이너에서 작업을 실행합니다. 연결할 컨테이너 이름을 지정하며, 연결할 사용자 이름을 접두사로 사용하고 구분 기호 "@"를 사용할 수 있습니다. 컨테이너 이름 대신 특수 문자 ".host"를 사용하면 로컬 시스템에 연결됩니다 (특정 사용자의 사용자 버스에 연결하는 데 유용합니다: "--user --machine=lennart@.host"). "@" 구문을 사용하지 않으면 root 사용자로 연결됩니다. "@" 구문을 사용하면 왼쪽 또는 오른쪽 측면을 생략할 수 있지만 (둘 다는 안 됩니다), 이 경우 로컬 사용자 이름과 ".host"가 암시됩니다.

     -C, --capsule=
         캡슐에서 작업을 실행합니다. 연결할 캡슐 이름을 지정합니다. 캡슐에 대한 자세한 내용은 capsule@.service(5)를 참조하십시오.

         버전 256에서 추가되었습니다.

     --no-ask-password
         특권 작업에 대해 사용자에게 인증을 요청하지 않습니다.

     --no-pager
         출력을 페이저로 파이프하지 않습니다.

     --legend=BOOL

     -h, --help
         간단한 도움말 텍스트를 출력하고 종료합니다.

     --version
         간단한 버전 문자열을 출력하고 종료합니다.

EXIT STATUS
     성공 시 0이 반환되며, 그 외에는 0이 아닌 실패 코드가 반환됩니다.

     systemctl은 LSB 3.0.0[3]에 정의된 LSB가 정의한 반환 코드를 사용합니다.

     Table 5. LSB 반환 코드
     ┌───────┬──────────────────────┬──────────────────────┐
     │ Value │ Description in LSB   │ Use in systemd       │
     ├───────┼──────────────────────┼──────────────────────┤
     │ 0     │ "program is  running │ unit is active       │
     │       │ or service is OK"    │                      │
     ├───────┼──────────────────────┼──────────────────────┤
     │ 1     │ "program is dead and │ unit   not    failed │
     │       │ /var/run   pid  file │ (used by is-failed)  │
     │       │ exists"              │                      │
     ├───────┼──────────────────────┼──────────────────────┤
     │ 2     │ "program is dead and │ unused               │
     │       │ /var/lock  lock file │                      │
     │       │ exists"              │                      │
     ├───────┼──────────────────────┼──────────────────────┤
     │ 3     │ "program is not run‐ │ unit is not active   │
     │       │ ning"                │                      │
     ├───────┼──────────────────────┼──────────────────────┤
     │ 4     │ "program or  service │ no such unit         │
     │       │ status is unknown"   │                      │
     └───────┴──────────────────────┴──────────────────────┘

     LSB 서비스 상태와 systemd 유닛 상태 간의 매핑은 완벽하지 않으므로, 해당 반환 값에 의존하기보다는 특정 유닛 상태 및 하위 상태를 찾는 것이 더 좋습니다.

ENVIRONMENT
     $SYSTEMD_EDITOR
         유닛을 편집할 때 사용할 편집기입니다. $EDITOR 및 $VISUAL을 재정의합니다. $SYSTEMD_EDITOR, $EDITOR, $VISUAL 중 어느 것도 없거나 빈 문자열로 설정되었거나 실행에 실패한 경우, systemctl은 다음 순서로 잘 알려진 편집기를 실행하려고 시도합니다: editor(1), nano(1), vim(1), vi(1).

         버전 218에서 추가되었습니다.

     $SYSTEMD_LOG_LEVEL
         방출되는 메시지의 최대 로그 레벨입니다 (더 높은 로그 레벨, 즉 덜 중요한 메시지는 억제됩니다). 쉼표로 구분된 값 목록을 받습니다. 값은 (중요도 감소 순서로) emerg, alert, crit, err, warning, notice, info, debug 중 하나이거나 0...7 범위의 정수일 수 있습니다. 자세한 내용은 syslog(3)를 참조하십시오. 각 값은 해당 특정 로그 대상을 위한 최대 로그 레벨을 설정하기 위해 console, syslog, kmsg 또는 journal 중 하나로 접두사 지정된 후 콜론을 붙일 수 있습니다 (예: SYSTEMD_LOG_LEVEL=debug,console:info는 콘솔에 로깅하는 경우 info 레벨이어야 하는 것을 제외하고 debug 레벨로 로깅하도록 지정합니다). 전역 최대 로그 레벨이 대상별 최대 로그 레벨보다 우선한다는 점에 유의하십시오.

     $SYSTEMD_LOG_COLOR
         불리언 값입니다. true인 경우, tty에 작성된 메시지는 우선순위에 따라 색상이 지정됩니다.

         이 설정은 메시지가 터미널에 직접 작성될 때만 유용합니다. journalctl(1) 및 로그를 표시하는 다른 도구는 자체적으로 로그 레벨에 따라 메시지에 색상을 지정하기 때문입니다.

     $SYSTEMD_LOG_TIME
         불리언 값입니다. true인 경우, 콘솔 로그 메시지에 타임스탬프가 접두사로 지정됩니다.

         이 설정은 메시지가 터미널이나 파일에 직접 작성될 때만 유용합니다. journalctl(1) 및 로그를 표시하는 다른 도구는 자체적으로 항목 메타데이터를 기반으로 타임스탬프를 첨부하기 때문입니다.

     $SYSTEMD_LOG_LOCATION
         불리언 값입니다. true인 경우, 메시지는 메시지가 발생하는 소스 코드의 파일 이름과 줄 번호로 접두사 지정됩니다.

         로그 위치는 어쨌든 journal 항목에 메타데이터로 첨부되는 경우가 많다는 점에 유의하십시오. 이를 메시지 텍스트에 직접 포함하는 것은 프로그램 디버깅 시 편리할 수 있습니다.

     $SYSTEMD_LOG_TARGET
         로그 메시지의 목적지입니다. console (첨부된 tty에 로깅), console-prefixed (첨부된 tty에 로깅하지만 로그 레벨과 "facility"를 인코딩하는 접두사를 사용, syslog(3) 참조), kmsg (커널 순환 로그 버퍼에 로깅), journal (저널에 로깅), journal-or-kmsg (사용 가능한 경우 저널에, 그렇지 않으면 kmsg에 로깅), auto (적절한 로그 대상을 자동으로 결정, 기본값), null (로그 출력 비활성화) 중 하나입니다.

     $SYSTEMD_PAGER, $PAGER
         --no-pager가 지정되지 않은 경우 사용할 페이저입니다. $SYSTEMD_PAGER가 설정되어 있으면 그것이 사용되며, 그렇지 않으면 $PAGER가 사용됩니다. $SYSTEMD_PAGER와 $PAGER 둘 다 설정되지 않은 경우, less(1) 및 more(1)을 포함하여 잘 알려진 페이저 구현 세트를 차례로 시도하여 하나를 찾습니다. 페이저 구현을 찾지 못하면 페이저가 호출되지 않습니다. 해당 환경 변수를 빈 문자열 또는 "cat" 값으로 설정하는 것은 --no-pager를 전달하는 것과 동일합니다.

         참고: $SYSTEMD_PAGERSECURE가 설정되어 있지 않으면, $SYSTEMD_PAGER와 $PAGER는 페이저를 비활성화하는 데 ( "cat" 또는 ""로)만 사용될 수 있으며, 그 외에는 무시됩니다.

     $SYSTEMD_LESS
         less에 전달되는 옵션을 재정의합니다 (기본값은 "FRSXMK").

         사용자는 특히 두 가지 옵션을 변경하고 싶을 수 있습니다:

         K
             이 옵션은 Ctrl+C가 눌렸을 때 페이저가 즉시 종료하도록 지시합니다. less가 자체적으로 Ctrl+C를 처리하여 페이저 명령 프롬프트로 다시 전환할 수 있도록 하려면 이 옵션을 해제하십시오.

             $SYSTEMD_LESS의 값이 "K"를 포함하지 않고 호출된 페이저가 less인 경우, Ctrl+C는 실행 파일에 의해 무시되며 페이저에 의해 처리되어야 합니다.

         X
             이 옵션은 페이저가 termcap 초기화 및 비초기화 문자열을 터미널로 전송하지 않도록 지시합니다. 이는 페이저가 종료된 후에도 명령 출력이 터미널에 계속 표시되도록 기본적으로 설정됩니다. 그럼에도 불구하고, 이는 일부 페이저 기능이 작동하지 않도록 방지하며, 특히 마우스로 페이지 출력을 스크롤할 수 없습니다.

         일반 $LESS 환경 변수를 설정하는 것은 systemd 도구에 의한 less 호출에는 영향을 미치지 않는다는 점에 유의하십시오.

         더 자세한 논의는 less(1)를 참조하십시오.

     $SYSTEMD_LESSCHARSET
         less에 전달되는 문자 집합을 재정의합니다 (기본값은 "utf-8"이며, 호출 터미널이 UTF-8 호환으로 결정된 경우).

         일반적인 $LESSCHARSET 환경 변수를 설정하는 것은 systemd 도구에 의한 less 호출에는 영향을 미치지 않는다는 점에 유의하십시오.

     $SYSTEMD_PAGERSECURE
         less(1)와 같은 일반적인 페이지 매김 명령어는 "페이지 매김", 즉 출력 스크롤링 외에도 다른 파일 열기 또는 쓰기 및 임의의 셸 명령 실행을 지원합니다. 명령이 sudo(8) 또는 pkexec(1)과 같은 높은 권한으로 호출될 때, 페이지 매김기는 보안 경계가 됩니다. 엄격하게 제한된 기능만을 가진 프로그램이 페이지 매김기로 사용되도록 주의해야 하며, 새 파일 열기나 생성 또는 서브프로세스 시작과 같은 의도하지 않은 대화형 기능은 허용되지 않습니다. 페이지 매김기가 이를 지원하는 경우, 아래에 설명된 대로 페이지 매김기의 "보안 모드"를 활성화할 수 있습니다 (대부분의 페이지 매김기는 이를 고려하여 작성되지 않았습니다). 신뢰할 수 없는 사용자가 높은 권한으로 명령을 실행하도록 허용할 때는 "보안 모드"를 명시적으로 활성화하거나 --no-pager 또는 PAGER=cat을 사용하여 페이지 매김기를 완전히 비활성화하는 것이 좋습니다.

         이 옵션은 부울 인수를 받습니다. true로 설정되면 페이지 매김기의 "보안 모드"가 활성화됩니다. "보안 모드"에서는 페이지 매김기를 호출할 때 LESSSECURE=1이 설정되며, 이는 페이지 매김기에게 새 파일을 열거나 생성하거나 새 서브프로세스를 시작하는 명령을 비활성화하도록 지시합니다. 현재 이 변수를 이해하고 "보안 모드"를 구현하는 것으로 알려진 것은 less(1)뿐입니다.

         false로 설정되면 페이지 매김기에 제한이 가해지지 않습니다. SYSTEMD_PAGERSECURE=0을 설정하거나 상속된 환경에서 이를 제거하지 않으면 사용자가 임의의 명령을 호출할 수 있게 될 수 있습니다.

         $SYSTEMD_PAGERSECURE가 설정되지 않은 경우, systemd 도구는 "보안 모드"가 활성화되어야 하는지 그리고 페이지 매김기가 이를 지원하는지 자동으로 파악하려고 시도합니다. 효과적인 UID가 로그인 세션 소유자와 같지 않은 경우 (geteuid(2) 및 sd_pid_get_owner_uid(3) 참조) 또는 sudo(8) 또는 유사한 도구($SUDO_UID가 설정된 경우 [4])에서 실행되는 경우 "보안 모드"가 활성화됩니다. 그러한 경우, SYSTEMD_PAGERSECURE=1이 설정되며 "보안 모드"를 구현하는 것으로 알려지지 않은 페이지 매김기는 전혀 사용되지 않습니다. 이 자동 감지는 권한을 상승시키는 가장 일반적인 메커니즘만을 다루며 편의를 위한 것임을 유의하십시오. $SYSTEMD_PAGERSECURE를 명시적으로 설정하거나 페이지 매김기를 비활성화하는 것이 권장됩니다.

         $SYSTEMD_PAGER 또는 $PAGER 변수가 페이지 매김기를 비활성화하는 것 외에 존중되려면, $SYSTEMD_PAGERSECURE도 설정되어야 합니다.

     $SYSTEMD_COLORS
         부울 인수 또는 특수 값을 받습니다. 기본값(설정되지 않음)으로, systemd 및 관련 유틸리티는 가능한 경우 출력에 색상을 사용합니다. $COLORTERM이 "truecolor" 또는 "24bit"로 설정된 경우 24비트 색상이 활성화되며, 그렇지 않은 경우 256색이 활성화됩니다. 단, $NO_COLOR 또는 $TERM이 색상이 비활성화되었음을 나타내는 경우는 제외합니다.

         true
             설정되지 않은 경우와 동일하지만 $NO_COLOR는 무시됩니다.

         false
             출력은 단색이 됩니다.

         "16", "256", "24bit"
             각각 기본 16 ANSI 색상, 256색 또는 24비트 색상을 항상 사용합니다.

         "auto-16", "auto-256", "auto-24bit"
             $TERM 및 콘솔이 연결된 환경에 따라 주어진 수의 색상을 사용합니다.

     $SYSTEMD_URLIFY
         값은 부울이어야 합니다. 이 기능을 지원하는 터미널 에뮬레이터의 출력에 클릭 가능한 링크가 생성될지 여부를 제어합니다. 이는 $TERM 및 기타 조건에 따라 systemd가 내리는 결정을 재정의하기 위해 지정될 수 있습니다.
